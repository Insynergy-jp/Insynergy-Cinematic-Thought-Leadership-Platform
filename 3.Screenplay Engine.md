# Purpose

This section defines the qualitative and quantitative objectives of the Screenplay Engine within the Cinematic Thought Leadership Platform.

The Screenplay Engine transforms the abstract **Story Model** into a cinematic screenplay suitable for visual production. Its purpose is treated as a **first-class architectural requirement**, equal in importance to story quality, visual quality, and system reliability.

The engine's purpose is **not** to create dialogue-heavy scripts. It exists to define, in a form that downstream visual planning can consume without ambiguity:

* what happens
* who acts
* where it happens
* why the audience should care
* how the story progresses visually

The screenplay becomes the **authoritative source** for all downstream visual planning. Every optimization, validation, and export defined in Part 3 shall serve this contract without compromising narrative integrity.

---

# Design Philosophy

The Screenplay Engine adopts the following principle:

> **The screenplay is the contract, not the article.**

Accordingly, screenplay generation shall prioritize:

1. Dramatizing the Story Model rather than explaining the source article
2. Expressing meaning through observable action and visual subtext
3. Reusing validated Story Artifacts rather than reinterpreting source material
4. Enforcing exactly one dramatic purpose and one conflict per scene
5. Deferring conceptual exposition until the audience is dramatically prepared

Volume of dialogue is considered a liability, not a measure of quality.

---

# Primary Objectives

The Screenplay Engine shall satisfy the following goals.

| Objective                     | Target        | Priority |
| ----------------------------- | ------------- | -------- |
| Screenplay Generation Time    | ≤ 30 sec      | High     |
| Scene Count per Screenplay    | 8–12 scenes   | Critical |
| Dramatic Purpose per Scene    | Exactly 1     | Critical |
| Dominant Conflict per Scene   | Exactly 1     | Critical |
| Dialogue Words per Line       | ≤ 15 words    | High     |
| Dialogue Lines per Turn       | ≤ 2 lines     | High     |
| Scene Duration                | 4 / 7 / 10 sec (min / target / max) | High |
| Fountain + JSON Export        | Both required | Critical |
| Continuity Validation         | Must pass     | Critical |

The target environment assumes the same CI/CD runner as the rest of the pipeline, consuming immutable Story Artifacts produced by the Story Engine.

---

# Success Definition

Screenplay generation is considered successful only if all of the following conditions are met:

* Every story is transformed into a screenplay of 8–12 ordered scenes.
* Every scene has exactly one dramatic purpose and one dominant conflict.
* Every character appearing in a scene pursues an objective.
* Dialogue creates dramatic tension rather than exposition.
* All actions are visually observable.
* The screenplay exports successfully to **both** Fountain and JSON.
* Continuity validation passes without violation.

Meeting the generation-time target alone does not constitute success.

---

# Position in the Pipeline

The Screenplay Engine occupies a fixed position between the Story Engine and the Shot Planner.

```text
Story Engine
      │
      ▼
Screenplay Engine
      │
      ▼
Shot Planner
      │
      ▼
Storyboard
```

The Screenplay Engine consumes **only Story Artifacts**. It shall never re-read the original article. This isolation is normative and is enforced by the constraints defined below.

---

# Inputs

The engine receives immutable Story artifacts as its sole input contract.

```yaml
inputs:

  dramatic-premise.json

  theme.json

  logline.json

  character-bible.yaml

  story-arc.json

  three-act-structure.json
```

No input outside this set is permitted. The original article is explicitly excluded from the input contract.

---

# Outputs

The engine produces the following artifacts. These files are immutable once approved.

```text
screenplay.fountain

screenplay.json

scene-index.json

dialogue.json

continuity.json
```

Downstream modules shall consume only these screenplay artifacts and shall not reach back into Story Artifacts or source material.

* Action Ratio

---

# Screenplay Constraints

The following constraints are mandatory.

## Constraint 1 — Story First

No optimization may reduce narrative quality.

Example:

Reducing the number of scenes solely to decrease generation time is prohibited unless approved through the Screenplay Quality Gate.

---

## Constraint 2 — Article Isolation

The engine shall never re-read the original article.

The screenplay is derived exclusively from immutable Story Artifacts. Any dependency on source material is a validation failure.

---

## Constraint 3 — Immutability

Approved screenplay artifacts shall never be mutated in place.

A material change in Story Artifacts produces a new screenplay; it does not edit an existing one.

---

## Constraint 4 — Determinism

Repeated generation from identical Story Artifacts and identical engine configuration shall produce an identical screenplay.

Random scene ordering or non-deterministic dialogue selection is prohibited.

---

## Constraint 5 — Singular Intent

Every scene shall express exactly one dramatic purpose and one dominant conflict.

Scenes that carry multiple purposes or multiple conflicts fail validation.

---

# Generation Priorities

The screenplay generation order is fixed.

```text
1. Establish Scene Purpose

↓

2. Assign Character Objectives & Conflict

↓

3. Write Observable Action

↓

4. Add Dialogue Only Where Tension Requires It

↓

5. Encode Visual Subtext & Transitions
```

This ordering is normative and shall guide all future screenplay-generation improvements.

# 3.1.2 Screenplay Architectural Principles

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.0

---

# Purpose

This section defines the permanent architectural principles governing the Screenplay Engine.

These principles are normative. Every scene generator, dialogue generator, validator, and exporter shall conform to them. Any deviation requires an Architecture Decision Record (ADR) documenting the exception.

The principles exist to ensure that the screenplay remains a **deterministic, immutable, dramatically disciplined contract** derived exclusively from Story Artifacts.

---

# Principle 1 — Story First

No optimization may reduce narrative quality.

Every generation decision must answer the following questions before it is adopted:

* Does this preserve the dramatic premise?
* Does this serve the audience's emotional experience?
* Does this respect the approved three-act structure?
* Can it be validated by the Screenplay Quality Gate?

If these questions cannot be answered, the decision shall not proceed.

---

# Principle 2 — Article Isolation

The engine derives the screenplay exclusively from immutable Story Artifacts.

```text
Story Artifacts

↓

Screenplay Engine

↓

Screenplay
```

The original article shall never be read by the Screenplay Engine. Any dependency on source material is a validation failure.

---

# Principle 3 — Immutable Artifacts

Approved screenplay artifacts are immutable.

```text
Story

↓

Screenplay

↓

Approved
```

Once approved, `screenplay.fountain`, `screenplay.json`, `scene-index.json`, `dialogue.json`, and `continuity.json` must never change. A material change in Story Artifacts produces a new screenplay; it never edits an existing one.

Benefits include:

* deterministic downstream planning
* reproducibility
* cache stability
* resume capability

---

# Principle 4 — Singular Purpose

Every scene exists for exactly one dramatic reason.

Exactly one purpose is allowed per scene. Scenes carrying multiple purposes fail validation.

Concepts belong to narration or visual metaphor — not to scene purpose.

---

# Principle 5 — Singular Conflict

Each scene contains exactly one dominant conflict.

A scene may reference tension from other forces, but only one dominant conflict may drive it. Multiple dominant conflicts in a single scene fail validation.

---

# Principle 6 — Observable Action Only

Action shall be written visually.

Forbidden:

```text
Maya thinks the governance architecture is insufficient.
```

Correct:

```text
Maya reaches for the emergency override.

The system rejects her credentials.
```

Thinking is not cinematic. Actions must be observable.

---

# Principle 7 — Dialogue Creates Tension

Dialogue exists only to create dramatic tension.

Dialogue must never explain the article. Exposition disguised as dialogue is prohibited.

Correct dialogue expresses a question, demand, refusal, warning, decision, or revelation — never a definition.

---

# Principle 8 — Silence Is Dialogue

Silence is treated as dialogue.

A deliberate absence of speech is a valid and encouraged dramatic device where it increases tension. Silence shall be represented explicitly rather than omitted.

---

# Principle 9 — Concepts Belong to Act 3

Conceptual meaning is restricted.

```text
Act 1 → dramatize
Act 2 → escalate
Act 3 → reveal concept
```

Concepts may appear only in Act 3, and only through dramatized understanding — never through Act 1 explanation.

---

# Principle 10 — Continuity Is Correctness

The screenplay maintains continuity across character, wardrobe, location, time, emotion, countdown, and props.

Any continuity violation fails validation. Continuity is treated as a correctness property, not a stylistic preference.

---

# Principle 11 — Determinism

Repeated generation from identical Story Artifacts and identical engine configuration shall produce an identical screenplay.

Random scene ordering, non-deterministic dialogue selection, or unstable scene identifiers are prohibited.

---

# Principle 12 — Emotion Must Evolve

Each scene records the protagonist's emotional state before and after the scene.

Emotion must evolve continuously across the screenplay. A flat emotional trajectory fails the intent of the dramatic premise.

---

# Principle 13 — Every Character Has an Objective

Every character appearing in a scene must pursue an objective.

Characters without objectives are prohibited. An objectiveless character is a validation failure.

---

# Principle 14 — Canonical Export

Every screenplay exports to both Fountain and JSON.

No custom syntax is permitted. Fountain output must be standard Fountain; JSON output must conform to the canonical schema.

---

# Principle 15 — Quality Gate Has Priority

The Screenplay Quality Gate has final authority.

An invalid screenplay shall never advance to Shot Planning, regardless of generation-time pressure. Quality has higher priority than speed.

---

# Architectural Decision Matrix

| Decision            | Rule                                   |
| ------------------- | -------------------------------------- |
| Source Input        | Story Artifacts only                   |
| Original Article    | Never read                             |
| Screenplay Artifact | Immutable after approval               |
| Scene Purpose       | Exactly one                            |
| Scene Conflict      | Exactly one dominant                   |
| Action              | Observable only                        |
| Dialogue            | Tension only, never exposition         |
| Concepts            | Act 3 only                             |
| Continuity          | Validated as correctness               |
| Export              | Fountain + JSON, no custom syntax      |
| Generation          | Deterministic                          |
| Quality             | Higher priority than speed             |

# 3.1.3 Canonical Screenplay & Scene Structure

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.0

---

# Purpose

This section defines the canonical structure of a screenplay and the internal structure of a scene.

The structure is normative. Every generated screenplay shall conform to the act model, scene model, and heading model defined here. Downstream Shot Planning depends on this structure being stable and unambiguous.

---

# Design Philosophy

The screenplay adopts the following principle:

> **Structure is the guarantee that meaning survives production.**

Accordingly, structure shall be:

1. Ordered — every scene has a fixed position
2. Bounded — every scene belongs to exactly one act
3. Complete — every scene contains all required fields
4. Uniform — every scene heading follows one grammar
5. Purposeful — every scene declares exactly one dramatic purpose

---

# Canonical Screenplay Structure

Each screenplay consists of ordered scenes distributed across three acts.

```text
ACT 1

Scene 1
Scene 2
Scene 3

ACT 2

Scene 4
Scene 5
Scene 6
Scene 7

ACT 3

Scene 8
Scene 9
Scene 10
```

The scene count shall remain within the range of 8–12 scenes.

---

# Act Boundaries

No scene may belong to multiple acts.

```text
Scene → exactly one Act
```

Act membership is fixed at generation time and is immutable once approved. A scene's act assignment shall be derived from the approved three-act structure.

---

# Scene Structure

Each scene contains the following fields.

```yaml
scene:

  heading

  purpose

  characters

  objective

  conflict

  action

  dialogue

  visual_subtext

  transition
```

A scene missing any required field fails validation.

---

# Scene Heading

Every scene begins with a standard screenplay heading.

Example

```text
INT. GLOBAL ENTERPRISE OPERATIONS CENTER - NIGHT
```

The heading is the authoritative declaration of where and when the scene occurs.

---

# Heading Components

Every heading decomposes into three components.

```yaml
heading:

  interior_exterior

  location

  time
```

---

# Allowed Interior/Exterior Values

Only the following values are permitted.

```yaml
INT

EXT

INT./EXT.
```

No other spatial qualifier is allowed.

---

# Dramatic Purpose

Every scene exists for one reason. Exactly one purpose is allowed.

Allowed purposes

```yaml
Introduce protagonist

Reveal hidden risk

Increase urgency

Create reversal

Force decision

Resolve conflict
```

---

# Forbidden Purposes

The following are prohibited as scene purposes.

```yaml
Explain concept

Show architecture

Describe framework
```

Concepts belong to narration or visual metaphor — not scene purpose.

---

# Scene Ordering Rules

Scene order is deterministic and normative.

```yaml
ordering:

  scenes_are_ordered: true

  scene_belongs_to_one_act: true

  act_order_fixed: true

  scene_ids_stable: true
```

Reordering a scene requires regenerating the screenplay from changed Story Artifacts.

---

# Acceptance Criteria

Section 3.1.3 is complete when:

* The canonical three-act, 8–12 scene structure is defined.
* Every scene belongs to exactly one act.
* The required scene field set is documented and enforced.
* Scene headings follow a single, uniform grammar with allowed INT/EXT values.
* Exactly one dramatic purpose is permitted per scene, with forbidden purposes explicitly listed.
* Scene ordering is deterministic and stable.

# 3.1.4 Dramatic System — Objectives, Conflict, Subtext & Emotion

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.0

---

# Purpose

This section defines the dramatic machinery of a scene: character objectives, conflict, visual subtext, and emotional state.

Together these fields ensure that every scene contains a force, an opposition, a hidden meaning, and an evolving emotional trajectory. They are the source from which Shot Planning derives dramatic emphasis.

---

# Design Philosophy

The dramatic system adopts the following principle:

> **Drama is objective meeting obstacle.**

Accordingly, every scene shall express:

1. what each character wants (objective)
2. what stands against it (conflict)
3. what it secretly means (visual subtext)
4. how it changes the protagonist (emotion)

A scene lacking any of these is dramatically inert and fails validation.

---

# Character Objective

Every character appearing in a scene must pursue an objective.

Example

```yaml
MAYA:
  stop_transaction

LEGAL_DIRECTOR:
  verify_authority

AI_AGENT:
  continue_execution
```

Characters without objectives are prohibited.

---

# Objective Rules

Objectives are constrained as follows.

```yaml
objective_rules:

  every_present_character_has_objective: true

  objective_is_actionable: true

  objective_is_scene_scoped: true
```

An objective must be pursuable through observable action within the scene.

---

# Conflict

Each scene contains one dominant conflict.

Types

```yaml
Human vs Machine

Human vs Institution

Human vs Time

Human vs Self

Institution vs Reality
```

---

# One Conflict Rule

Only one dominant conflict is permitted per scene.

```text
Scene → exactly one dominant conflict
```

Secondary tensions may exist, but exactly one conflict shall dominate the scene. Multiple dominant conflicts fail validation.

---

# Visual Subtext

Every scene contains hidden meaning.

Example

Scene

```text
The emergency override fails.
```

Visual Subtext

```yaml
Responsibility exists.

Authority does not.
```

This subtext guides storyboard generation and shall be recorded explicitly for every scene.

---

# Emotional State

Each scene records the protagonist's emotional state before and after the scene.

Example

```yaml
before:
  focused

after:
  unsettled
```

---

# Emotion Evolution Rule

Emotion must evolve continuously.

```yaml
emotion_rules:

  before_and_after_recorded: true

  emotion_changes_across_screenplay: true

  trajectory_is_continuous: true
```

A flat emotional trajectory across the screenplay contradicts the dramatic premise and fails the intent of the Quality Gate.

---

# Acceptance Criteria

Section 3.1.4 is complete when:

* Every character appearing in a scene has an actionable objective.
* Each scene declares exactly one dominant conflict from the allowed conflict types.
* Every scene records explicit visual subtext for downstream storyboard generation.
* Each scene records protagonist emotion before and after.
* Emotion is required to evolve continuously across the screenplay.

# Purpose

This section defines how action and dialogue are written within a scene.

Action and dialogue are the observable surface of the screenplay. Their discipline determines whether the screenplay dramatizes the Story Model or degrades into exposition. This section establishes the rules that keep both cinematic.

---

# Design Philosophy

The action and dialogue system adopts the following principle:

> **Show the pressure; never narrate the idea.**

Accordingly:

1. Action shall be observable
2. Dialogue shall create tension
3. Exposition shall be prohibited
4. Silence shall be treated as dialogue

---

# Action Writing

Action is written visually.

Good

```text
Maya reaches for the emergency override.

The system rejects her credentials.

The countdown continues.
```

Bad

```text
Maya thinks the governance architecture is insufficient.
```

Thinking is not cinematic. Actions must be observable.

---

# Action Rules

Action is constrained as follows.

```yaml
action_rules:

  observable_only: true

  no_internal_thought: true

  present_tense: true

  concrete_and_physical: true
```

---

# Dialogue Purpose

Dialogue exists only to create dramatic tension.

Dialogue must never explain the article.

Wrong

```text
Decision Design allocates institutional authority.
```

Correct

```text
MAYA

Who decided
that the agent could approve this?
```

---

# Dialogue Rules

Dialogue is bounded by explicit limits.

```yaml
words_per_line: 15

lines_per_turn: 2
```

Dialogue exceeding these limits fails validation.

---

# Dialogue Categories

Dialogue shall express one of the following.

```yaml
Question

Demand

Refusal

Warning

Decision

Revelation
```

Exposition is prohibited. Any line that defines, describes, or explains a concept is rejected.

---

# Silence

Silence is treated as dialogue.

Example

```text
LEGAL DIRECTOR

...

MAYA realizes
there is no answer.
```

Silence is encouraged where dramatic and shall be represented explicitly rather than omitted.

---

# Acceptance Criteria

Section 3.1.5 is complete when:

* Action is written as observable, present-tense, physical events with no internal thought.
* Dialogue exists solely to create dramatic tension and never explains the article.
* Dialogue respects the limits of 15 words per line and 2 lines per turn.
* Every line of dialogue falls within an allowed dramatic category.
* Exposition is prohibited and rejected by validation.
* Silence is treated and represented as a valid form of dialogue.

# 3.1.8 Continuity, Quality Gate & Metrics

# Purpose

This section defines the correctness controls of the Screenplay Engine: continuity validation, the Screenplay Quality Gate, and the metrics that quantify screenplay health.

Together they determine whether a screenplay is permitted to advance to Shot Planning. They are the enforcement boundary that guarantees only valid screenplays reach production.

---

# Design Philosophy

The correctness system adopts the following principle:

> **A screenplay is not done until it is proven consistent.**

Accordingly:

1. Continuity is validated as a correctness property
2. The Quality Gate has final authority
3. Metrics make screenplay health measurable

---

# Continuity Rules

The screenplay maintains continuity across the following dimensions.

```yaml
continuity_checks:

  Character

  Wardrobe

  Location

  Time

  Emotion

  Countdown

  Props
```

Any violation fails validation.

---

# Continuity Validation

Continuity is enforced, not advisory.

```yaml
continuity_rules:

  all_dimensions_checked: true

Cache Key match

Skip screenplay generation

↓

Implemented

↓

Superseded

↓

Archived
```

# ADR Format

```yaml

```

---

# ADR-301

## Screenplay Consumes Only Story Artifacts

### Status

Accepted

### Context

The original article is unstructured and interpretive. Re-reading it during screenplay generation would reintroduce ambiguity and break determinism.

### Decision

The Screenplay Engine consumes only immutable Story Artifacts and never reads the original article.

### Consequences

Benefits

* deterministic screenplays
* stable cache keys
* clean separation of concerns

Trade-offs

* Story Artifacts must be complete and authoritative.

---

# ADR-302

## Screenplay is Immutable After Approval

### Status

Accepted

### Context

Downstream Shot Planning and Storyboard generation depend on a stable screenplay.

### Decision

Approved screenplay artifacts are immutable. A material change in Story Artifacts produces a new screenplay rather than editing an existing one.

### Consequences

Benefits

* reproducible downstream planning
* cache stability
* resumable execution

---

# ADR-303

## Exactly One Dramatic Purpose per Scene

### Status

Accepted

### Context

Scenes carrying multiple purposes dilute dramatic clarity and complicate shot planning.

### Decision

Every scene declares exactly one dramatic purpose. Multiple purposes fail validation.

---

# ADR-304

## Exactly One Dominant Conflict per Scene

### Status

Accepted

### Context

Multiple dominant conflicts fragment a scene's dramatic focus.

### Decision

Each scene contains exactly one dominant conflict drawn from the allowed conflict types.

---

# ADR-305

## Dialogue Creates Tension, Never Exposition

### Status

Accepted

### Context

Exposition disguised as dialogue turns cinematic scenes into lectures.

### Decision

Dialogue exists solely to create dramatic tension and must never explain the article. Exposition is prohibited.

---

# ADR-306

## Actions Must Be Observable

### Status

Accepted

### Context

Internal thought cannot be rendered by visual production.

### Decision

Action is written as observable, physical events. Internal thought is prohibited.

---

# ADR-307

## Concepts Are Restricted to Act 3

### Status

Accepted

### Context

Introducing concepts before the audience is dramatically prepared reduces them to explanation.

### Decision

Concepts appear only in Act 3 and only through dramatized understanding.

---

# ADR-308

## Canonical Export to Fountain and JSON

### Status

Accepted

### Context

Human reviewers and machine planners require different faithful representations.

### Decision

Every screenplay exports to both standard Fountain and canonical JSON, with no custom syntax.

---

# ADR-309

## Continuity is Part of Correctness

### Status

Accepted

### Context

Continuity errors propagate into storyboard and render, where they are expensive to fix.

### Decision

Continuity is validated as a correctness property across character, wardrobe, location, time, emotion, countdown, and props. Any violation fails validation.

---

# ADR-310

## Screenplay Generation is Deterministic

### Status

Accepted

### Context

Non-deterministic generation would break caching, resume, and reproducibility.

### Decision

Repeated generation from identical Story Artifacts and configuration produces an identical screenplay.

# 3.1.11 Operational State Model

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.0

---

# Purpose

This section defines the operational state model of the Screenplay Engine.

It describes the lifecycle of a screenplay from request through approval and export, the states of an individual scene, the ownership of state, and the invariants that govern legal transitions. Downstream orchestration relies on these states to schedule, resume, and gate work.

---

# Design Philosophy

The state model adopts the following principle:

> **Every screenplay has exactly one well-defined state at every moment.**

Accordingly:

1. States are explicit and enumerable
2. Transitions are validated
3. Human approval is a first-class state
4. State is persisted for resume

---

# Screenplay Lifecycle

A screenplay progresses through the following states.

```text
Pending

↓

Generating

↓

Generated

↓

Validating

↓

Validated

↓

AwaitingApproval

↓

Approved

↓

Exported
```

A failure at any stage transitions to a terminal failure state.

---

# Screenplay State Definitions

```yaml
states:

  Pending:
    Story Artifacts available; generation not started

  Generating:
    scenes and dialogue being produced

  Generated:
    draft screenplay complete

  Validating:
    continuity and Quality Gate running

  Validated:
    all checks passed

  AwaitingApproval:
    pending human approval

  Approved:
    immutable; ready for export

  Exported:
    artifacts written and immutable

  Failed:
    validation or generation error
```

---

# Scene Lifecycle

Each scene carries its own state within a screenplay.

```text
Drafted

↓

Populated

↓

Validated

↓

Locked
```

A scene becomes `Locked` only when its parent screenplay is approved.

---

# State Ownership

Each state is owned by exactly one component.

```yaml
ownership:

  Generating: SceneGenerator / DialogueGenerator

  Validating: ContinuityValidator / ScreenplayQualityGate

  AwaitingApproval: Human Reviewer

  Exported: ScreenplayExporter
```

No component may transition a state it does not own.

---

# State Transition Rules

Transitions are constrained.

```yaml
transition_rules:

  forward_only_until_failure: true

  no_skipping_validation: true

  approval_required_before_export: true

  approved_is_immutable: true
```

Any illegal transition is rejected.

---

# Human Approval States

Human approval is an explicit, blocking state.

```text
Validated

↓

AwaitingApproval

↓ (human approves)

Approved
```

Export shall not begin before approval. A rejected screenplay returns to regeneration from changed Story Artifacts, never in-place edit.

---

# State Validation

State integrity is enforced.

```yaml
state_validation:

  single_active_state: true

  transitions_logged: true

  invalid_transition_rejected: true
```

---

# State Persistence

State is persisted to support resume.

```yaml
persistence:

  state_recorded: true

  resumable: true

  snapshot_with_config: true
```

A build interrupted after validation resumes at `AwaitingApproval` without regenerating validated work.

---

# Operational Invariants

The following invariants always hold.

```yaml
invariants:

  one_state_per_screenplay: true

  validation_precedes_approval: true

  approval_precedes_export: true

  approved_artifacts_immutable: true

  failure_is_terminal_until_regeneration: true
```

---

# Operational Interfaces

State is exposed through stable interfaces.

```python
ScreenplayStateMachine

SceneStateTracker

ApprovalGate

StatePersistence
```

Each interface must be independently testable.

---

# State Metrics

Operational dashboards report the following.

```yaml
state_metrics:

  time_in_generating

  time_in_validating

  time_awaiting_approval

  regeneration_count

  approval_turnaround
```

---

# Acceptance Criteria

Section 3.1.11 is complete when:

* The screenplay lifecycle and scene lifecycle are enumerated.
* Every state has a single owning component.
* Transition rules forbid skipping validation and require approval before export.
* Human approval is modeled as an explicit blocking state.
* State is persisted to support resume, and operational invariants are defined.
