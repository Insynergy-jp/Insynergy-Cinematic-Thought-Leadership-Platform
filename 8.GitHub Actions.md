# 8.0.1 Specification v2.1 Amendment — Agent Review Workflow

## Canonical GitHub Actions projection

Agent Review Mode shall be projected as a separate secret-bearing job between deterministic planning and the protected rendering environment:

```text
plan job                       no provider secrets
  -> agent-review job          environment: planning-ai
  -> render execution job      environment: render-approval (required reviewers)
  -> compose and final gates
  -> publish job               environment: publication-approval (required reviewers)
```

The v2.1 canonical Environment names are `planning-ai`, `render-approval`, and `publication-approval`. Earlier examples such as `production-execute` describe the same pre-render approval role; new or updated workflows shall use `render-approval`.

## Workflow shape

`plan.yml` remains deterministic and publishes sealed planning artifacts plus the Execution Plan. It shall not receive `OPENAI_API_KEY`, `RUNWAY_API_KEY`, or publication credentials.

The repository shall add a reusable `_agent-review.yml` workflow, or an equivalently isolated `agent-review` job, with this contract:

```yaml
on:
  workflow_call:
    inputs:
      build_id: {type: string, required: true}
      review_mode: {type: string, required: true}
      execution_plan_artifact: {type: string, required: true}
    secrets:
      OPENAI_API_KEY: {required: false}
    outputs:
      review_status: {}
      review_key: {}
      review_report_content_hash: {}
      review_report_artifact: {}
```

Required job behavior is:

1. download and hash-verify the sealed planning bundle;
2. resolve and validate `off` or `review` mode;
3. in `off`, emit `NOT_APPLICABLE` without resolving a secret or importing a live provider;
4. in `review`, target Environment `planning-ai`, resolve `OPENAI_API_KEY`, enforce input and output budgets, and run the orchestrator's `agent-review` verb;
5. validate `agent-review-report.json` against Part 9 and resolve every evidence pointer;
6. upload the immutable report and a safe summary, then persist its reference and hash in the manifest;
7. expose only safe scalar outputs to downstream jobs.

The workflow file shall not contain review policy, prompt text, model-selection logic, report-repair logic, or approval logic. Those contracts belong to versioned application code and configuration.

## Environment and secret matrix

| Environment / job | Required protection | Allowed secrets | Prohibited secrets |
| --- | --- | --- | --- |
| planning job (no Environment) | normal branch/workflow policy | none | `OPENAI_API_KEY`, `RUNWAY_API_KEY`, publication credentials |
| `planning-ai` | trusted branches and actors; no fork access | `OPENAI_API_KEY` | `RUNWAY_API_KEY`, publication credentials |
| `render-approval` | required reviewers, deployment-branch restriction, timeout policy | `RUNWAY_API_KEY` and only required render credentials | `OPENAI_API_KEY`, publication credentials |
| `publication-approval` | required reviewers, deployment-branch restriction | only required publication credentials | `OPENAI_API_KEY`, `RUNWAY_API_KEY` unless a separately documented publication need exists |

`OPENAI_API_KEY` shall be a GitHub **Environment secret** on `planning-ai`, not a repository variable, workflow input, command argument, artifact, cache entry, job output, or checked-in `.env` value. Non-secret agent settings shall use allow-listed GitHub Environment or repository variables.

## Configuration mapping

| Specification setting | GitHub Actions source |
| --- | --- |
| `INSYNERGY_AGENT_REVIEW_MODE` | explicit workflow input resolved against an allow-listed repository/environment variable |
| `OPENAI_MODEL_REVIEW` | non-secret variable on `planning-ai` |
| `OPENAI_REASONING_EFFORT` | non-secret variable on `planning-ai` |
| `OPENAI_TRACE_MODE` | non-secret variable on `planning-ai` |
| `AGENT_REVIEW_TIMEOUT_SECONDS` | versioned config or allow-listed variable |
| `AGENT_REVIEW_MAX_INPUT_BYTES` | versioned config or allow-listed variable |
| `AGENT_REVIEW_MAX_OUTPUT_TOKENS` | versioned config or allow-listed variable |
| `OPENAI_API_KEY` | Environment secret on `planning-ai` |

Unknown, empty, or policy-disallowed values fail preflight. The workflow shall record the effective non-secret configuration and resolved model identifier in the report and job summary.

## Approval binding and reviewer surface

Before the first job that targets `render-approval`, the workflow summary shall expose safe links or artifact references for:

* build ID and Execution Plan content hash;
* Agent Review disposition and report content hash;
* blocking findings, invariant codes, and evidence locations;
* deterministic planning gate summary;
* estimated rendering cost and shot count;
* any review error or human-exception requirement.

The execution preflight shall compare the approved Execution Plan hash and Agent Review report hash with the downloaded artifacts. GitHub Environment approval alone is not sufficient if either hash is missing or mismatched. Any regenerated report or plan requires a new `render-approval` decision.

`publication-approval` remains a separate post-rendering gate and shall not be satisfied by Agent Review or `render-approval`.

## Fork, pull request, and untrusted event policy

Fork pull requests and other untrusted events shall never receive `OPENAI_API_KEY`. Their required checks run with `review_mode=off` and a fake provider for contract tests. Live Agent Review is allowed only from trusted events and branches selected by policy. `pull_request_target` shall not check out or execute untrusted code with the `planning-ai` Environment.

## Artifacts, retention, and logs

The immutable review report shall be uploaded under a deterministic name containing the build ID and review-key prefix. The report is approval evidence and shall use the retention and access policy of planning artifacts. GitHub job outputs and summaries may contain status, hashes, codes, safe recommendations, usage totals, and artifact links; they shall not contain the Article body, full prompt, full raw response, credentials, or chain-of-thought.

Caches shall be keyed by the full review key and shall never include secrets. A cache hit must still validate schema and content hash. Actions artifacts are a transport and retention surface; the platform artifact store and manifest remain authoritative.

## Failure and concurrency behavior

The job shall use a concurrency group derived from the full review key so duplicate trusted events do not create duplicate live submissions. Cancellation, timeout, authentication failure, rate limit, quota failure, invalid output, and provider failure shall emit classified safe summaries and keep all downstream provider-spending jobs blocked until remediation or a Part 7-permitted, attributable human exception. Structural failures such as a missing report, invalid schema, unresolved evidence, failed deterministic gate, or hash mismatch remain non-waivable. Re-run shall reconcile existing immutable evidence before attempting another call.

## Required workflow tests

* workflow syntax and policy validation;
* `off` mode with no OpenAI secret;
* fake-provider structured success and manual-review outcomes;
* missing `OPENAI_API_KEY` in live `review` mode;
* fork and `pull_request_target` secret-isolation tests;
* plan/report hash mismatch blocking `render-approval` execution;
* exact review-key cache hit with zero provider invocations;
* report regeneration invalidating approval;
* log and artifact secret scanning;
* confirmation that `RUNWAY_API_KEY` is unavailable to `planning-ai` and `OPENAI_API_KEY` is unavailable to render and publish jobs.

Live OpenAI calls shall not be part of required pull-request CI. A manually triggered trusted smoke workflow may validate the real integration under `planning-ai`, bounded by an explicit model and budget.

## ADRs

| ADR | Decision |
| --- | --- |

| ADR-8.0.1-B | `OPENAI_API_KEY` is scoped to the `planning-ai` Environment. |

## Acceptance criteria

This amendment is complete when `off` runs without an OpenAI secret, `review` can access the secret only through `planning-ai`, untrusted events cannot reach it, report artifacts are schema/hash verified, provider-spending jobs remain blocked until remediation or an allowed recorded exception, structural failures remain non-waivable, the approved hash pair is checked before Runway contact, and publication still requires its own protected Environment.

# About This Document

Part 8 defines how the Cinematic Thought Leadership Platform **executes on GitHub Actions** as its primary Continuous Integration / Continuous Delivery (CI/CD) execution environment.

Part 8 is **not** a general GitHub Actions tutorial. It is a platform-specific execution specification. It takes the domain model established in Parts 5 and 6 — the Build Orchestrator, the Execution Plan, Worker Pools, the Render Queue, the Render Manifest, the Checkpoint & Resume Engine, and the Provider Task model — and defines exactly how those constructs are **mapped onto, hosted by, and constrained by** the GitHub Actions runtime.

The governing thesis of Part 8 is:

> **GitHub Actions is a runtime host, not the domain. Domain contracts defined in Parts 5 and 6 remain authoritative; GitHub Actions provides ephemeral, stateless execution surfaces onto which those contracts are projected without alteration.**

Everything in Part 8 must be implementable by Codex directly from this specification, and must preserve every invariant established in Parts 5 and 6 — identity, determinism, idempotency, integrity, isolation, cost-awareness, resumability, and governance.

# Purpose

This section fixes the **boundary** between the platform's execution domain (Parts 5 and 6) and its **runtime host** (GitHub Actions). It answers one question precisely: *what does Part 8 own, and what does it merely host?*

The platform's correctness — narrative integrity, deterministic planning, idempotent rendering, resumable builds, cost governance — is defined entirely in Parts 1–6. GitHub Actions contributes **none** of these guarantees. It contributes only an execution surface: a way to trigger work, schedule jobs onto runners, pass artifacts between jobs, hold secrets, and gate on human approval.

Part 8 exists so that this execution surface is used in a way that **preserves** every domain guarantee, and **introduces no new authority** of its own. When Part 8 and the domain appear to conflict (for example, GitHub's automatic retry of a job versus the domain's idempotent retry classification), the domain wins and Part 8 specifies how to make GitHub's behavior conform.

---

# Governing Principle

> **GitHub Actions is where the platform runs, not what the platform is. The host schedules; the domain decides.**

Three consequences are normative:

1. **No domain truth originates on a runner.** Build state, task state, provider task identity, cache eligibility, and quality verdicts are read from and written to durable stores defined in Parts 5 and 6. A runner is a transient computer that borrows this truth, acts, persists the result, and is discarded.

2. **No domain rule is re-implemented in YAML.** Workflow YAML expresses *scheduling* (what runs, where, in what order, with what concurrency). It never expresses *policy* (whether a retry is allowed, whether a shot is READY, whether a build may compose). Policy lives in the orchestrator invoked *by* the steps.

3. **The host is replaceable.** The same domain must be runnable on a developer laptop (Part 6 local profile) or a future distributed runtime without changing a single domain contract. Part 8 is one binding of many; it must not leak GitHub-specific assumptions into the domain.

---

* secret handling, permission scoping, and supply-chain integrity on Actions (8.5);
* cache/artifact backends and minutes/cost enforcement on Actions (8.6);
* observability surfaces (summaries, annotations, required checks) and their coupling to Quality Gates (8.7);
* retry/timeout/cancellation *mechanics* on Actions, and operational runbooks (8.8).

# What Part 8 Does Not Own

* narrative, story, screenplay, shot, or storyboard logic (Parts 2–4);
* render strategy, prompt assembly, provider API semantics, validation criteria (Part 5);
* the build state machine, task state machine, retry classification, dependency semantics, cache invalidation logic, or resume decision-making (Part 6);
* any definition of "correct output." Part 8 can make a build *run*; only the domain can make it *right*.

---

# The Execution Boundary Contract

Every crossing of the boundary obeys four rules:

| Rule | Statement |
| ---- | --------- |
| B-1 Read-then-act | A runner reads authoritative state before acting; it never assumes state from a previous job's memory. |
| B-2 Persist-before-ack | A runner persists its result to the durable store before signaling success to the host (8.4.2). |
| B-3 No host-only truth | If a fact exists only in host state (a job output, a run annotation, a runner file), it is not authoritative and must be reconstructable from the domain store. |
| B-4 Host-agnostic contracts | Nothing a runner writes to the domain encodes a GitHub identifier as domain identity (no `run_id` as build identity, no `run_attempt` as task attempt). |

These four rules are the load-bearing content of Part 8. Every later section is an application of them.

---

# Reference Configuration

```yaml
execution_boundary:
  authority: domain           # Parts 5/6 are authoritative
  host: github_actions
  host_role: scheduler_only
  truth_on_runner: prohibited # B-3
  read_then_act: required     # B-1
  persist_before_ack: required# B-2
  host_ids_as_domain_ids: prohibited  # B-4
  replaceable_host: true      # local & distributed bindings must remain possible
```

Configuration is validated at orchestrator startup on every runner; a runner that detects a boundary violation in its own configuration (e.g. instructed to treat `run_id` as `build_id`) fails closed before acting.

---

# Security Requirements

* The boundary is also a trust boundary: a runner is treated as **partially untrusted** compute. Nothing a runner produces is authoritative until validated and persisted through domain contracts (mirrors Part 6.3.9 WD-308).
* No domain secret is materialized on a runner beyond the lifetime of the step that needs it (8.5).
* Host-provided identifiers are never used to authorize domain actions.

---

# Testing Requirements

## Unit Tests

* a runner refuses to start when configured to treat a host identifier as domain identity (B-4);
* `persist_before_ack` is enforced: a simulated ack before persist is rejected.

## Integration Tests

* a job reads current build state from the domain store, not from a prior job's output, and behaves identically whether or not the prior job's outputs are present;
* the same orchestrator command produces identical domain effects when run locally (Part 6 local profile) and on a runner — proving host-agnosticism.

## Failure Injection Tests

* kill a runner after persist but before ack → replay is a no-op (B-2);
* kill a runner after act but before persist → the domain shows no effect and the work is safely re-attempted.

---

# Quality Gate

Section 8.1.1's boundary is satisfied only when: no authoritative fact is discoverable solely in host state; every runner obeys read-then-act and persist-before-ack; and an identical domain result is demonstrable across the local and Actions bindings.

---

# Architectural Invariants

## Invariant GA-101
GitHub Actions holds no authoritative domain state; all such state is reconstructable from Part 5/6 durable stores.

## Invariant GA-102
Workflow YAML expresses scheduling only; no domain policy is implemented in YAML.

## Invariant GA-103
A host identifier (`run_id`, `run_attempt`, job id) is never used as a domain identity or as an authorization token for a domain action.

## Invariant GA-104
Every boundary crossing obeys read-then-act (B-1) and persist-before-ack (B-2).

## Invariant GA-105
The domain remains runnable on non-Actions hosts without contract change; Part 8 introduces no irreversible coupling.

# 8.1.2 — GitHub Actions Execution Principles

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section states the normative principles that govern **every** workflow, job, and step in Part 8. They are to Part 8 what the fifteen Architectural Performance Principles (Part 6.1.2) are to Part 6: a fixed set of rules that any later design must satisfy and against which any workflow may be audited.

---

# Governing Principle

> **Design workflows so that the cheapest correct execution is also the default one.** Every principle below removes a class of waste, risk, or non-determinism before optimization is even considered.

---

# The Principles

## Principle 1 — Host, Not Domain
Workflows schedule; the orchestrator decides (8.1.1). No policy in YAML.

## Principle 2 — Ephemerality by Assumption
Every runner is assumed fresh and discarded. No step relies on state left by a previous run, a warm cache, or a persistent filesystem. State that must survive lives in the domain store (8.4).

## Principle 3 — Manifest Is External
The Render/Runtime Manifest is never authored on a runner as its home; it is loaded, mutated through domain contracts, and persisted externally (8.4.2). Loss of all runners loses no truth.

## Principle 4 — Submit, Don't Hold
No runner blocks on external provider wall-clock. Submission persists provider task identity and releases; monitoring and download are separate jobs (8.2.6, 6.3.9).

## Principle 5 — Fail Closed
Any ambiguity — missing input, invalid config, unverified approval, budget breach, integrity mismatch — halts the affected work rather than proceeding under unsafe assumptions.

## Principle 6 — Cost-Gated Fan-Out
No high-cost work (provider rendering) begins until an estimate is computed and checked against the minutes/provider budget, with human approval above threshold (8.6.4, Part 6 Principles 11–12).

## Principle 7 — Resume by Re-run
Recovery is a re-run driven by the manifest, never a bespoke manual repair. Re-running is always safe because of idempotency (8.4.4–8.4.5).

## Principle 8 — Idempotent by Cache Key
Every externally-visible action (provider submission, CAS write, publish) is idempotent, keyed on content, so that re-runs and duplicate deliveries cause no double effect (8.4.5, 5.6.3).

## Principle 9 — Least Privilege
Each job requests the minimum `permissions:` and the minimum secrets, scoped to the narrowest environment (8.5). Default token is read-only; write is opt-in per job.

## Principle 10 — Pin Everything
Third-party actions are pinned to an immutable commit SHA; nothing runs from a mutable tag (8.5.5).

## Principle 11 — Deterministic Scheduling Inputs
Matrix sets, shard assignments, and job graphs are deterministic functions of domain state, not of time or run number (8.2.6, GA-263).

## Principle 12 — Observable, Not Chatty
Every job emits a structured summary and correlates its logs to the manifest; verbosity is bounded and secrets never appear in logs (8.7).

## Principle 13 — Concurrency Is Backpressure
Concurrency groups and `max-parallel` exist to protect providers and budget, not merely to speed things up (8.6.5, 6.3.16).

## Principle 14 — Single-Crossing Composition
Composition happens at most once per build, behind an all-or-hold barrier; no partial trailer is ever produced (8.2.7, Part 5).

## Principle 15 — Quality Gates Are Blocking
A failed Quality Gate is a required-check failure that blocks composition/publish; gates are never advisory (8.7.4, 6.3.24).

# 8.1.3 — Execution Topology & Runner Model

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the physical and logical topology through which the platform executes on GitHub Actions: the containment hierarchy (repository → workflow → job → step → runner), the runner classes, and the ephemeral lifecycle every runner obeys. It gives later sections a fixed vocabulary for "where does this run."

---

# Governing Principle

> **A runner is a disposable, single-use computer.** It is provisioned, does one job, persists its results to the domain, and is destroyed. Nothing about its identity, filesystem, or memory is durable.

---

# Architectural Position

```text
Repository
  └── Workflow (a .yml file, triggered by an event)
        └── Job (runs on exactly one runner)
              └── Step (a shell command or an action)
                    └── Runner process (ephemeral host)
```

A **workflow run** corresponds to a Build generation (or a coordinated segment of one). A **job** is the unit of runner allocation and the natural home for a Worker-Pool activity (8.3.4). A **step** is the unit of orchestrator invocation. The **runner** is the ephemeral compute below the boundary (8.1.1).

---

# Runner Classes

| Class | Provisioning | Used for | Notes |
| ----- | ------------ | -------- | ----- |
| GitHub-hosted | On-demand, GitHub-managed | Planning, approval bookkeeping, light coordination | Clean image, per-job VM, metered minutes. |
| Self-hosted (CPU) | Platform-managed fleet | FFmpeg composition, packaging, validation, download | Sized for memory/disk; autoscaled (8.3.5). |
| Self-hosted (GPU/local-gen) | Platform-managed fleet | Local generative steps (if any), heavy asset ops | Capability-labeled (8.3.3). |
| Self-hosted (network-egress) | Restricted fleet | Provider submission/monitor/download | Egress to provider APIs only; least privilege (8.5). |

Runner class is selected by `runs-on` labels, which resolve to the Worker capability matrix (8.3.3).

---

# Ephemeral Lifecycle

```text
PROVISION → REGISTER → LEASE JOB → RUN STEPS → PERSIST → DEREGISTER → DESTROY
```

Normative properties:

* **Single-use.** A self-hosted runner processes one job then is recycled (ephemeral runners), preventing state bleed between builds (Principle 2).
* **Job-scoped filesystem.** The workspace exists only for the job; anything needed later is persisted to artifacts/CAS/manifest before the job ends (8.4).
* **No cross-job memory.** Job B never reads job A's process memory or local disk; it reads the domain store (B-1).
* **Drainable.** On cancel/timeout the runner receives SIGTERM and must checkpoint in-flight work to the manifest before exit (8.3.6, 8.8.3).

---

# Topology Constraints

| Constraint | Value | Source |
| ---------- | ----- | ------ |
| Max jobs per matrix | 256 | GitHub platform |
| Max concurrent jobs | Plan-dependent + self-hosted fleet size | GitHub + fleet (8.3.5) |
| Job max wall-clock | `timeout-minutes` (bounded per pool) | 8.8.3 |
| Artifact size/retention | Configurable retention window | 8.6.3 |
| Cache scope | Per repo/branch, key-addressed | 8.6.2 |

These constraints are inputs to sharding (8.2.6), capacity planning (8.3.5), and cost control (8.6).

---

# Reference Configuration

```yaml
topology:
  hosted_runners:
    use_for: [plan, approve-bookkeeping, light-coordination]
  self_hosted_pools:
    cpu:     { labels: [self-hosted, cpu],     ephemeral: true }
    ffmpeg:  { labels: [self-hosted, ffmpeg],  ephemeral: true, min_mem_gb: 16 }
    egress:  { labels: [self-hosted, egress],  ephemeral: true, egress: providers_only }
  runner_lifecycle:
    ephemeral: true
    single_use: true
    drain_on_signal: SIGTERM
    checkpoint_before_exit: true
```

Invalid topology (e.g. a non-ephemeral runner assigned to a submission pool) fails closed at fleet registration.

---

# Security Requirements

* Self-hosted runners are network-restricted to what their pool needs (egress runners reach provider APIs; FFmpeg runners need none).
* Ephemerality is mandatory for any runner that handles secrets, preventing secret persistence across jobs.
* No runner is shared across repositories or trust levels.

---

# Testing Requirements

## Unit Tests
* label→pool resolution is total and deterministic;
* a non-ephemeral runner is rejected from secret-handling pools.

## Integration Tests
* job B reads state from the domain and succeeds even when job A's workspace is unavailable (no cross-job memory);
* SIGTERM to a running job triggers a manifest checkpoint before exit.

## Recovery Tests
* destroy a runner mid-job → the build resumes from the manifest on re-run with no lost truth.

---

# Quality Gate

Topology passes when every job's runner class is capability-matched, every secret-handling runner is ephemeral, and cross-job independence is demonstrated.

---

# Architectural Invariants

## Invariant GA-121
Every runner is single-use and job-scoped; no state survives a runner beyond what is persisted to the domain.

## Invariant GA-122
A job never depends on another job's runner memory or local filesystem.

## Invariant GA-123
Runner class is resolved from capability labels matching the Worker capability matrix (6.3.9).

## Invariant GA-124
Secret-handling runners are ephemeral and network-restricted to their pool's needs.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-920 | The containment hierarchy repo→workflow→job→step→runner is canonical; jobs are the unit of pool allocation. |
| ADR-921 | Self-hosted runners are ephemeral and single-use to prevent state bleed. |
| ADR-922 | Runner classes are network-restricted per pool need. |
| ADR-923 | The 256-leg matrix limit and per-pool timeouts are first-class topology constraints feeding sharding and capacity planning. |

---

# Acceptance Criteria

Section **8.1.3** is complete when:

* the containment hierarchy and runner classes are defined;
* the ephemeral lifecycle and its normative properties are specified;
* topology constraints feeding later sections are tabulated;
* invariants GA-121…GA-124 and ADRs ADR-920…ADR-923 are recorded.

# 8.2.1 — Purpose & Workflow Authoring Boundary

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section fixes what a workflow author may and may not decide. It draws, within the host, a second boundary: between **scheduling** (the workflow's job) and **governing** (the orchestrator's job). It ensures every workflow in 8.2 is a thin scheduler over a thick, host-agnostic orchestrator.

---

# Governing Principle

> **Workflows schedule; the orchestrator governs.** A workflow decides *when and where* a step runs; the orchestrator decides *whether the work is valid, allowed, and complete.*

---

# Architectural Position

```text
Trigger/event ─► Workflow (scheduling)
                    │ needs / matrix / concurrency / timeout / env
                    ▼
                  Step ─► orchestrator <verb> (governing)
                             │ reads/writes manifest, applies policy
                             ▼
                          Domain effect (persisted)
```

The workflow arranges steps; each step invokes an orchestrator verb (`plan`, `fanout-plan`, `render-submit`, `render-monitor`, `render-download`, `compose`, `package`, `publish`, `resume`). All policy is inside those verbs.

---

# What a Workflow May Decide

* which trigger/event starts it and how the event resolves to a profile (8.1.6);
* the job graph (`needs`), matrix expansion, concurrency groups, and timeouts;
* which runner class each job uses (`runs-on` labels);
* which environment (and thus which secrets/approvals) a job runs under;
* what artifacts/cache a job restores and saves.

# What a Workflow May Not Decide

* whether a task may retry, or how a failure is classified (6.3.13 → orchestrator);
* whether a shot is READY, or a build may compose (6.3.24 / Part 5 → orchestrator);
* the identity or attempt number of a task (6.3.5 → manifest);
* cache eligibility or invalidation (6.2 → orchestrator);
* provider submission idempotency (5.6.3 → orchestrator).

Any `if:` expression that encodes such a decision is an anti-pattern (AP-8-002).

---

# Reference Configuration

```yaml
authoring_boundary:
  workflow_decides: [trigger, profile_resolution, job_graph, matrix, concurrency, timeout, runner_class, environment, artifact_cache_io]
  orchestrator_decides: [validity, retry_eligibility, readiness, identity, cache_eligibility, idempotency, completion]
  policy_in_yaml: prohibited
```

---

# Security Requirements

* Environment selection (which grants secrets) is a scheduling decision, but *access* to a secret is gated by GitHub environment protection, not by workflow logic (8.5.2).
* A workflow may not construct provider credentials; it may only select the environment that provides them via OIDC.

---

# Testing Requirements

## Unit Tests
* workflow-lint flags any `if:`/step that encodes retry, readiness, or identity decisions;
* every scheduling knob used is on the allowed list.

## Integration Tests
* moving a policy decision from the orchestrator into YAML causes a lint failure and is rejected.

---

# Quality Gate

The authoring boundary passes when every workflow decides only scheduling concerns and every governing decision is delegated to an orchestrator verb.

---

# Architectural Invariants

## Invariant GA-201
Workflows express scheduling only; all governing decisions are delegated to orchestrator verbs.

## Invariant GA-202
Task identity and attempt number originate in the manifest, never in workflow expressions.

## Invariant GA-203
No `if:` expression encodes retry-eligibility, readiness, cache-eligibility, or idempotency.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-970 | A scheduling/governing boundary is enforced inside every workflow. |
| ADR-971 | Orchestrator verbs are the sole home of policy; workflows call them. |
| ADR-972 | Policy-bearing `if:` expressions are prohibited and lint-detectable. |

---

# Acceptance Criteria

# 8.2.2 — Trigger & Event Model

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines every event that can start platform work, how each resolves to a Build Profile, and the guarantees around idempotent, non-duplicating triggering.

---

# Governing Principle

> **An event proposes work; the orchestrator decides whether that work already exists.** Triggering is never assumed to be exactly-once; idempotency makes duplicate triggers harmless.

---

# Event Catalogue

| Event | Meaning | Resolves to | Notes |
| ----- | ------- | ----------- | ----- |
| `workflow_dispatch` | Operator starts a build | explicit profile input | Primary human entry (8.8.5). |
| `repository_dispatch` | System/API starts or continues a build | payload `type` | Cross-run continuation (8.4.6). |
| `push` | Content/config change | preview (or configured) | Optional; gated to avoid accidental cost. |
| `pull_request` | Proposed change | preview, no secrets | Fork-safe (8.5.6). |
| `schedule` | Periodic monitor/reconcile | monitor-only | Provider polling (8.2.6). |
| `workflow_call` | Reusable-workflow invocation | inherits caller profile | Internal chaining. |

---

# Event → Build Identity

A trigger carries or derives a `build_id`. The orchestrator, on receiving an event:

```text
1. Resolve build_id (from input, payload, or content hash).
2. Load manifest for build_id (B-1).
3. If build_id already terminal → no-op (idempotent trigger).
4. If build_id in-flight → attach/continue (8.4.6), not restart.
5. Else → create build generation.
```

A duplicate `repository_dispatch` for the same `build_id` therefore never starts a second build (Principle 8).

---

# Concurrency & De-duplication

```yaml
concurrency:
  group: build-${{ inputs.build_id }}
  cancel-in-progress: false     # never cancel an in-flight build via a duplicate trigger
```

The build-scoped concurrency group ensures a single active run per `build_id`; duplicate triggers queue or no-op rather than racing.

---

# Reference Configuration

```yaml
triggers:
  catalogue: [workflow_dispatch, repository_dispatch, push, pull_request, schedule, workflow_call]
  event_to_profile: deterministic       # 8.1.6
  build_id_resolution: [input, payload, content_hash]
  duplicate_trigger: idempotent_noop_or_continue
  concurrency_group: build-<build_id>
  cancel_in_progress: false
```

---

# Security Requirements

* `pull_request` from forks never receives secrets and is pinned to preview (8.5.6).
* `repository_dispatch` requires an authenticated caller with least-privilege token; its payload is validated before use.
* Event payloads are untrusted input; they are schema-validated and never `eval`-ed into shell (injection prevention, 8.5.6).

---

# Testing Requirements

## Unit Tests
* each event resolves to the correct profile; unknown event fails closed;
* duplicate `repository_dispatch` for a terminal `build_id` is a no-op;
* fork `pull_request` cannot obtain secrets or a production profile.

## Integration Tests
* two near-simultaneous triggers for one `build_id` yield a single active run (concurrency group);
* a `repository_dispatch` continuation attaches to an in-flight build rather than restarting it.

## Failure Injection Tests
* a malformed event payload is rejected by schema validation, not executed.

---

# Quality Gate

Triggering passes when every event deterministically resolves to a profile, duplicate triggers are idempotent, fork events are secret-free, and payloads are validated before use.

---

# Architectural Invariants

## Invariant GA-211

## Invariant GA-212

## Invariant GA-213

| --- | -------- |

```text
      ▼ (rejected/expired)
build held → MANUAL_REVIEW / CANCELLED
```

The gate is realized as a job in `execute.yml` (or a bridging `approve.yml`) that targets a protected environment; GitHub blocks the job until a required reviewer approves.

---

# What the Reviewer Approves

The approval surface presents, from the manifest and plan (never fabricated):

* the `build_id`, profile, and Execution Plan content hash;
* shot count and render set size after cache exclusion (8.2.6);
* the estimated provider spend and minutes (8.6.4);
* the budget caps and whether the estimate is within them.

Approval binds to the **plan content hash**; if the plan changes, prior approval is void and re-approval is required (GA-232).

---

# Gate Behavior

| Outcome | Effect |
| ------- | ------ |
| Approved | Execution proceeds; approval record (who/when/plan-hash) persisted to manifest. |
| Rejected | Build transitions to CANCELLED or MANUAL_REVIEW; no provider contacted. |
| Timed out | Gate expires; build held; re-trigger required. |
| Budget exceeded pre-approval | Gate is not even offered; planner failed closed (8.6.4). |


Preview/benchmark/debug profiles may skip the gate (8.1.6); production always requires it.

# 8.2.6 — Matrix Strategy & Shot-Level Render Fan-Out

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the platform fans a single approved **Execution Plan** out into **shot-level parallel render work** on GitHub Actions, using the workflow **matrix strategy** as the parallelization surface.

Shot rendering is the only massively parallel stage in the pipeline (Part 6.1.3, Stage S7; Part 5 Parallel Execution Model). Every other stage is either sequential and cheap (planning) or a single-crossing barrier (composition). The economic and latency behavior of the entire platform is therefore dominated by **how well shot rendering fans out** and **how little runner time that fan-out consumes**.

This section specifies:

* how the approved Execution Plan is expanded into a matrix of render Tasks;
* the **submit-then-release** discipline that prevents runners from being held while external providers render;
* the decoupling of **submission**, **monitoring**, and **download** so that GitHub Actions minutes are spent on coordination, not on waiting;
* how the platform respects GitHub's hard **256-job matrix limit** through deterministic sharding;
* how idempotency, manifest authority, cost gating, and resumability from Parts 5 and 6 are preserved across ephemeral matrix legs.

This section does **not** define provider API semantics (Part 5.3), render-task state ownership (Part 5.4.4), retry classification (Part 5.4.5 / 6.3.13), the composition barrier (Part 5 / 8.2.7), or manifest persistence mechanics (8.4.2). It consumes those contracts.

---

# Governing Principle

Shot-level fan-out on GitHub Actions is governed by one rule, inherited directly from Part 6.3.9 and Part 5:

> **Fan out on submission; release on persistence; join only at composition. A runner is never held for the duration of an external provider render.**

Two corollaries follow, both normative:

1. **The matrix parallelizes coordination, not waiting.** A matrix leg exists to *submit* a render and persist its provider task identity, or to *reconcile and download* a completed render — never to block on a provider's rendering wall-clock.

2. **The matrix is a projection of the manifest, not a source of truth.** The set of legs, their inputs, and their success/failure are derived from and written back to the Render Manifest (Part 5.4). If every matrix leg were lost mid-run, the identical remaining work would be reconstructable from the manifest (8.4.4).

A design that holds a runner in a polling loop for a multi-minute Runway render violates this principle and is prohibited (see Anti-Patterns below).

---

# Architectural Position

```text
Approved Execution Plan (artifact, immutable)              [8.2.4 / 8.2.5]
        │
        ▼
Fan-Out Planner (job: fanout-plan)
   • reads Execution Plan + Render Manifest
   • computes NOT-YET-READY shots (cache/incremental aware)   [6.2.6, 5.6.3]
   • estimates cost; enforces Minutes/Provider budget          [8.6.4]
   • emits shard descriptors as a matrix `include` set
        │
        ▼  (matrix, ≤256 legs per workflow)                    [GitHub limit]
┌───────────────────────────────────────────────────────────┐
│ Submission Matrix (job: render-submit)                     │
│   one leg per shard of shots                                │
│   • idempotent admission keyed on render cache key  [5.6.3] │
│   • submit to Provider Adapter (Runway/Motion/SVG/Type)     │
│   • persist provider_task_id to Manifest → RELEASE  [6.3.9] │
└───────────────────────────────────────────────────────────┘
        │  (runners released; providers render asynchronously)
        ▼
Monitor / Reconcile (job: render-monitor — poll, not hold)   [6.3.10]
   • decoupled poll of provider task status
   • advances Manifest state on completion/failure
        │
        ▼
Download & Validate Matrix (job: render-download)             [5.6, 6.3.9]
   • fetch completed assets; validate; register in CAS  [6.2.4]
   • mark shot READY in Manifest
        │
        ▼
All-or-Hold Barrier ───► Composition (single crossing)        [8.2.7, Part 5]
```

The Fan-Out Planner sits **downstream of approval** (8.2.4) and **upstream of the composition barrier** (8.2.7). It is the sole component that expands a Build into parallel legs. No other job may create render legs.

The submission matrix, monitor job, and download matrix are **three distinct jobs** precisely so that no single runner spans submission and completion. This decoupling is the GitHub Actions expression of the Provider Submission / Provider Monitor / Download Worker Pool separation defined in Part 6.3.9.

---

# Responsibilities

The fan-out subsystem is responsible for:

* deriving the set of render Tasks from the approved Execution Plan and the current Manifest;
* excluding shots already satisfied by cache hits or unchanged approved assets (incremental fan-out, Part 5.6.3 / 6.2.6);
* estimating render count and cost, and enforcing the Minutes/Provider budget **before** any provider is contacted (8.6.4);
* partitioning Tasks into **shards** that respect the 256-leg matrix limit and the per-provider concurrency limits (Part 5.4 / 6.3.16);
* emitting a deterministic matrix `include` set;
* performing **idempotent submission** so that a re-run never double-submits a render (8.4.5 / 5.4.1);
* persisting `provider_task_id` and authoritative Task state to the Manifest **before** acknowledging or releasing (write-before-ack, 8.4.2);
* releasing the runner immediately after provider task identity is durable;
* driving decoupled monitoring and download without holding submission capacity.

It is **not** responsible for:

* selecting render strategy or assembling prompts (Part 5.2 / 5.5);
* provider API invocation details (Part 5.3);
* render-task state ownership or the legal-transition guard (Part 5.4.4);
* retry eligibility classification (Part 5.4.5 / 6.3.13);
* asset validation outcomes (Part 5.6);
* composition or packaging (8.2.7);
* deciding creative fallbacks (escalated to Manual Review, Part 5).

---

# The Fan-Out Planner

The Fan-Out Planner is a single, cheap, deterministic job. It never contacts a provider.

## Inputs

```yaml
fanout_planner:
  inputs:
    execution_plan: execution-plan.json      # immutable, approved   [8.2.4]
    render_manifest: render_manifest.json     # authoritative state   [5.4.4]
    cache_index: cas-index.json               # CAS content index     [6.2.4]
  fail_closed_on:
    - execution_plan_unapproved
    - execution_plan_hash_mismatch
    - manifest_unreadable
```

## Algorithm

```text
1. Load approved Execution Plan; verify approval + content hash.       [fail closed]
2. Load Render Manifest (authoritative Task states).
3. Compute the render set:
     render_set = { shot ∈ plan.shots
                    | manifest.state(shot) ∉ {READY, CACHED, COMPOSED}
                      AND NOT cache_hit(shot)                          [5.6.3, 6.2.6]
                    }
4. If render_set is empty:
     emit empty matrix; signal "nothing to render"; proceed to barrier.
5. Estimate cost = Σ provider_cost(shot) for shot ∈ render_set.
6. Enforce budget:                                                    [8.6.4]
     if estimate > MINUTES_BUDGET or estimate > PROVIDER_SPEND_CAP:
        fail closed → require re-approval (Principle 12).
7. Partition render_set into shards (see Sharding).
8. Emit matrix `include` = [ shard descriptors ].
```

The planner is deterministic: given the same Execution Plan and Manifest, it emits the identical matrix. This is required for resumability (8.4.4) and for reproducible builds (Part 6.1.2).

## Output — Matrix Descriptor

```json
{
  "build_id": "20260720-001",
  "render_set_size": 47,
  "shard_count": 4,
  "matrix": {
    "include": [
      { "shard_id": "s0", "shots": ["sh-001","sh-002", "..."], "provider": "runway" },
      { "shard_id": "s1", "shots": ["sh-013","sh-014", "..."], "provider": "runway" },
      { "shard_id": "s2", "shots": ["sh-025","sh-026", "..."], "provider": "motion" },
      { "shard_id": "s3", "shots": ["sh-040","sh-041", "..."], "provider": "svg" }
    ]
  }
}
```

The descriptor is emitted as a job output and consumed by the downstream matrix via `fromJSON(...)`.

---

# Sharding

A **shard** is a deterministic partition of the render set assigned to a single matrix leg. Sharding exists to reconcile three constraints simultaneously:

| Constraint | Source | Consequence for sharding |
| ---------- | ------ | ------------------------ |
| ≤ 256 matrix legs per workflow | GitHub Actions platform limit | `shard_count ≤ 256`. |
| Per-provider concurrency limit | Part 5.4 / 6.3.16 | Concurrent legs per provider capped by `concurrency` group, not by leg count alone. |
| Cost/minutes budget | Principle 11 / 8.6.4 | Total submitted work bounded before fan-out. |

## Sharding Rules

1. **Provider affinity.** A shard contains shots for exactly one provider. Providers have different concurrency limits, rate limits, and adapters (Part 5.3); mixing them in one leg would couple their backpressure.

2. **Deterministic assignment.** Shot→shard assignment is a pure function of `sort(shot_id)` and `shard_count`. Re-running the planner yields identical shards. No randomness, no time-based assignment.

3. **Bounded leg count.** `shard_count = min(256, ceil(render_set_size / SHOTS_PER_SHARD), provider_leg_cap)`. Default `SHOTS_PER_SHARD` is profile-dependent (see Reference Configuration).

4. **Concurrency is enforced by groups, not leg count.** The matrix may declare more legs than may run at once; `concurrency` groups and `max-parallel` throttle actual simultaneity to respect provider limits (6.3.16). Leg count controls *coordination granularity*, concurrency controls *provider pressure*.

## Overflow Beyond 256

If `render_set_size` is so large that even at maximum `SHOTS_PER_SHARD` more than 256 legs would be required, the planner does **not** silently truncate. It:

* packs more shots per shard until `shard_count ≤ 256`; and if that still overflows,
* emits multiple **fan-out waves** — sequential matrix jobs, each ≤ 256 legs — correlated by `build_id`, and **logs the wave plan explicitly** (no silent capping, Part 6 observability principle).

A single trailer's shot count (Part 5: 8–12 scenes, shots per scene bounded) makes multi-wave fan-out rare in practice, but the specification defines it so the system fails loudly rather than dropping shots.

---

# Submission Matrix Leg (`render-submit`)

Each leg processes exactly one shard. A leg is short-lived: its wall-clock is submission + persistence, not rendering.

## Leg Contract

```text
For each shot in shard.shots (in deterministic order):

  1. Compute render cache key.                                  [5.6.3]
  2. Idempotent admission:                                      [5.4.1, 8.4.5]
       if manifest has an active or completed provider task
          for this cache key → SKIP (no re-submission).
  3. Resolve provider payload references (no bulk copy).        [Part 5 RIO]
  4. Submit to Provider Adapter.                                [5.3]
  5. On accepted submission:
       persist { shot, cache_key, provider_task_id, SUBMITTED } [write-before-ack]
       to the Render Manifest.                                  [8.4.2]
  6. RELEASE — do not poll for completion in this leg.          [6.3.9]

On leg crash before step 5 persists:
  the shot has no manifest record of submission → safe to re-submit
  (idempotency by cache key prevents a true double-charge if the
   provider did accept but the record was lost — reconciled in monitor).
```

## Critical Design Rule (inherited from 6.3.9 ADR-884)

> A submission leg **must not** hold its runner while the provider renders. It persists provider task identity and exits. Monitoring and downloading are separate jobs.

Holding a submission runner in a `sleep`/poll loop would:

* consume GitHub Actions minutes linearly with render wall-clock (cost violation, Principle 11);
* occupy a submission concurrency slot that another shard needs (throughput violation, 6.3.9);
* couple the leg's survival to the provider's latency (resilience violation).

Therefore the submission leg's success criterion is **"provider task identity is durably persisted,"** not **"asset is rendered."**

---

# Decoupled Monitoring & Download

## Monitor Job (`render-monitor`)

A separate job polls provider task status for all in-flight submissions of the build. It is **not** part of the submission matrix. It may be:

* a short poll job re-invoked on a `schedule`/`workflow_dispatch` cadence, or
* a bounded-duration poll loop with a hard `timeout-minutes` that re-queues itself via `repository_dispatch` if renders are still in flight (self-continuation, 8.4.6).

The monitor advances Manifest state (`SUBMITTED → RUNNING → DOWNLOADING` or `→ FAILED`) strictly through the legal-transition guard (Part 5.4.4). It never renders and never composes.

The choice of cadence vs. bounded-loop is a Build Profile decision (8.1.6): Preview builds favor a bounded loop for latency; Production builds may favor scheduled polling to minimize minutes. Both are equivalent with respect to correctness because state lives in the Manifest, not in the job.

## Download & Validate Matrix (`render-download`)

Once shots reach `DOWNLOADING`, a download matrix (sharded like submission) fetches assets, validates them (Part 5.6), registers passing assets in CAS (6.2.4), and marks shots `READY`. A failed validation routes to retry classification (Part 5.4.5 / 6.3.13), never to composition.

Download legs are also short-lived and stateless: their authority is the Manifest and CAS, not the runner filesystem.

# Example Workflow (Illustrative)

The following is a normative-by-shape example. Action versions are pinned to commit SHA in the real repository (8.5.5); placeholders are shown here for readability.

```yaml
name: render-fanout
on:
  workflow_call:
    inputs:
      build_id: { required: true, type: string }
      profile:  { required: true, type: string }   # preview | production | ...

permissions:
  contents: read
  id-token: write            # OIDC to provider/cloud; no long-lived keys   [8.5.2]

jobs:
  fanout-plan:
    runs-on: [self-hosted, control]                 # Control pool          [8.3.4]
    outputs:
      matrix: ${{ steps.plan.outputs.matrix }}
      render_count: ${{ steps.plan.outputs.render_count }}
    steps:
      - uses: actions/checkout@<pinned-sha>
      - id: plan
        run: orchestrator fanout-plan \
               --build-id "${{ inputs.build_id }}" \
               --profile  "${{ inputs.profile }}" \
               --emit-matrix          # deterministic; enforces budget       [8.6.4]

  render-submit:
    needs: fanout-plan
    if: ${{ needs.fanout-plan.outputs.render_count != '0' }}
    runs-on: [self-hosted, submission]              # Submission pool        [8.3.4]
    strategy:
      fail-fast: false                              # one shard's failure ≠ build failure
      max-parallel: 8                               # provider pressure guard [6.3.16]
      matrix: ${{ fromJSON(needs.fanout-plan.outputs.matrix) }}
    concurrency:
      group: submit-${{ inputs.build_id }}-${{ matrix.provider }}
      cancel-in-progress: false                     # never cancel a submit mid-persist
    timeout-minutes: 15                             # submit+persist only, never render
    steps:
      - uses: actions/checkout@<pinned-sha>
      - name: Submit shard (idempotent, submit-then-release)
        run: orchestrator render-submit \
               --build-id "${{ inputs.build_id }}" \
               --shard    "${{ matrix.shard_id }}"   # persists provider_task_id, exits [6.3.9]

  render-monitor:
    needs: render-submit
    runs-on: [self-hosted, monitor]                 # Monitor pool           [8.3.4]
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@<pinned-sha>
      - name: Poll & reconcile (decoupled; no render, no hold)
        run: orchestrator render-monitor \
               --build-id "${{ inputs.build_id }}" \
               --until-terminal-or-timeout          # re-dispatches if still in flight [8.4.6]

  render-download:
    needs: render-monitor
    runs-on: [self-hosted, download]                # Download pool          [8.3.4]
    strategy:
      fail-fast: false
      matrix: ${{ fromJSON(needs.fanout-plan.outputs.matrix) }}
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@<pinned-sha>
      - name: Download, validate, register in CAS, mark READY
        run: orchestrator render-download \
               --build-id "${{ inputs.build_id }}" \
               --shard    "${{ matrix.shard_id }}"   # [5.6, 6.2.4]
```

Key points the shape encodes: `fail-fast: false` (a shard failure isolates, per Part 5 task independence); `max-parallel` + `concurrency` as provider backpressure (6.3.16); submission `timeout-minutes` sized for *submission only*; monitoring and download as separate jobs; OIDC instead of static secrets (8.5.2).

# Security Requirements

* **No secrets in matrix descriptors.** The matrix `include` set is a job output visible in logs and to downstream jobs; it carries shot IDs and shard IDs only — never credentials, never prompt payloads, never provider tokens.
* **OIDC-first credentials.** Submission legs authenticate to providers via short-lived OIDC-federated credentials where the provider supports it (8.5.2 / 8.5.3); static provider keys, where unavoidable, are Environment-scoped secrets, never repository-wide.
* **Least-privilege token.** Fan-out jobs request `contents: read` and only the minimal `id-token: write`; no job in the fan-out path is granted `contents: write` or `packages: write`.
* **Untrusted output boundary.** A downloaded provider asset is untrusted until validated and CAS-registered (Part 6.3.9 Invariant WD-308); a matrix leg never treats a provider response as authoritative.
* **Fork isolation.** Fan-out never runs from a fork PR context with secrets (8.5.6); render execution is gated behind approval (8.2.4) which requires trusted-context reviewers.

---

# Testing Requirements

## Unit Tests

* deterministic sharding: identical (Execution Plan, Manifest) → identical matrix `include`;
* incremental exclusion: shots that are `READY`/`CACHED`/cache-hit are excluded from the render set (5.6.3);
* shard count respects `min(256, ceil(n/SHOTS_PER_SHARD), provider_leg_cap)`;
* provider affinity: no shard mixes providers;
* budget gate: estimate over cap fails closed and requires re-approval;
* empty render set emits an empty matrix and signals "nothing to render."

## Integration Tests

* full fan-out of a 47-shot plan across 4 shards → submit → monitor → download → all READY;
* submit-then-release: submission legs complete before providers finish rendering (assert leg wall-clock ≪ render wall-clock);
* monitor advances Manifest states through the legal-transition guard only;
* download validates, registers CAS objects, and marks READY; failed validation routes to retry classification, never to READY.

## Concurrency Tests

* `max-parallel` + `concurrency` groups cap simultaneous legs per provider at the configured limit;
* two legs racing on overlapping shots (misconfiguration) → idempotent admission ensures at most one submission per cache key;
* one shard failing (`fail-fast: false`) does not cancel sibling shards.

## Failure Injection Tests

* submission leg crashes **before** manifest persist → re-run re-submits; idempotency by cache key prevents a duplicate provider charge (reconciled by monitor);
* submission leg crashes **after** manifest persist → re-run observes existing provider task and SKIPs (no double submission, 8.4.5);
* provider accepts but the persistence write is lost → monitor reconciles the orphaned provider task against the manifest rather than fabricating or duplicating;
* matrix descriptor lost between plan and submit → re-run of the planner regenerates the identical matrix (8.4.4);
* render set > 256 legs at max packing → multi-wave plan emitted and logged; no shot dropped.

## Recovery Tests

* kill the entire fan-out run after submission; re-run reconstructs remaining work purely from the Manifest and completes without double submission;
* resume a build whose monitor timed out with renders still in flight → self-continuation re-attaches to existing provider tasks;
* total loss of `actions/cache` contents → CAS (authoritative) still yields cache hits; fan-out remains correct, only slower (8.6.2).

---

# Quality Gate

Fan-out passes its Quality Gate (feeding the build-level gate, 8.7.4 / 6.3.24) only when:

* every shot in the approved Execution Plan is either excluded by a valid cache hit or accounted for by exactly one shard;
* no submission leg held a runner for provider render wall-clock (audited from leg durations);
* no duplicate provider submission occurred for any cache key;
* the budget gate was evaluated before any provider was contacted;
* all READY transitions are backed by validated, CAS-registered assets;
* the barrier (8.2.7) receives a manifest in which every required shot is READY or the build is held for Manual Review — never a partial trailer.

A gate failure blocks composition (8.2.7) and, transitively, publication.

---

# Architectural Invariants

## Invariant GA-261

A matrix leg parallelizes coordination (submit / reconcile / download), never provider render wall-clock.

## Invariant GA-262

## Invariant GA-263

# Purpose

---

# Governing Principle

> **Compose once, only when whole.** The barrier is crossed at most once per build; a build with any required shot not READY never composes a partial trailer (Part 5, Principle 14).

---

# Architectural Position

```text
manifest: all required shots READY (validated, CAS-registered)
      │  compose.yml
      ▼

      ▼
job: ffmpeg-compose (self-hosted ffmpeg pool) → COMPOSING→COMPOSED
      ▼
job: package      → PACKAGING→PACKAGED (deliverable + metadata)
      ▼
```

```text
required = { shot ∈ plan | shot.required }
if ∀ s ∈ required : manifest.state(s) ∈ {READY, CACHED}:

else:
    hold build; route unmet shots to retry/Manual Review (Part 5.4.5)
```

* runs on the isolated FFmpeg self-hosted pool (8.3.4), sized for memory/disk (8.3.7);
* consumes only READY/CACHED assets resolved from CAS (never a runner's ad-hoc files);
* is deterministic given the same asset set and composition plan;
* writes the composed master and advances COMPOSING→COMPOSED with provenance (Part 5 manifest).

---

# Packaging Job

* produces the publishable package (renditions, thumbnails, captions, metadata.json projection);
* seals outputs (content-addressed, overwrite-prohibited, Part 5 RIO);
* advances PACKAGING→PACKAGED.

---

# Reference Configuration

```yaml
    transition: COMPOSING->COMPOSED
  packaging:
    seal_outputs: true
    content_addressed: true
    transition: PACKAGING->PACKAGED
```

# 8.2.9 — Job Dependency Graph, `needs` & Concurrency

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the Build dependency graph (Part 6.3.6) and scheduling/priority model (6.3.8) project onto GitHub `needs` edges and `concurrency` groups, and how host-level cancellation relates to domain-level build cancellation.

---

# Governing Principle

> **`needs` expresses ordering; `concurrency` expresses backpressure; neither expresses policy.** The orchestrator decides readiness; the graph merely arranges jobs consistent with those decisions.

---

# The `needs` DAG

```text
plan → preflight → fanout-plan → render-submit → render-monitor → render-download → barrier → compose → package → publish
                                     (matrix)                          (matrix)
```

Rules:

* `needs` edges are acyclic and mirror the domain dependency order (S1…S10);
* a downstream job starts only when its `needs` complete, but *readiness within* a job is an orchestrator decision (6.3.6);
* `fail-fast: false` on matrices preserves task independence (GA-270).

---

# Concurrency Groups

| Group | Purpose | `cancel-in-progress` |
| ----- | ------- | -------------------- |
| `build-<build_id>` | one active run per build (8.2.2) | false |
| `submit-<build_id>-<provider>` | provider backpressure (8.2.6) | false |
| `compose-<build_id>` | serialize composition | false |

`cancel-in-progress` is false for build-critical groups: a duplicate trigger must never cancel in-flight, persisted work.

---

# Host Cancellation vs Build Cancellation

| Event | Meaning | Handling |
| ----- | ------- | -------- |
| GitHub cancels a job (timeout/manual) | host eviction | drain + checkpoint to manifest (8.3.6, 8.8.3), then the build is RESUMABLE |
| Operator cancels the build (ops.yml) | domain decision | orchestrator transitions build to CANCELLED with compensation (6.3.14) |

A host cancellation never means the build is cancelled; it means the runner stopped and the build is resumable from the manifest.

---

# Reference Configuration

```yaml
graph:
  needs_dag: acyclic_mirrors_domain_order
  matrix_fail_fast: false
  concurrency:
    build: {group: build-<build_id>, cancel_in_progress: false}
    submit: {group: submit-<build_id>-<provider>, cancel_in_progress: false}
    compose: {group: compose-<build_id>, cancel_in_progress: false}
  host_cancel_means: drain_checkpoint_resumable
  build_cancel_means: orchestrator_transition_with_compensation
```

---

# Security Requirements

* Concurrency groups are keyed on `build_id`/provider, never on secrets or user-controlled free text (injection safety).

---

# Testing Requirements

## Unit Tests
* the `needs` DAG is acyclic and mirrors S1…S10;
* build-critical concurrency groups have `cancel-in-progress: false`.

## Integration Tests
* a duplicate build trigger does not cancel the in-flight run;
* a host job cancellation leaves the build RESUMABLE with a manifest checkpoint;
* an operator build-cancel transitions to CANCELLED with compensation.

## Failure Injection Tests
* cancel a matrix leg mid-submit → sibling legs unaffected (fail-fast false), build resumable.

---

# Quality Gate

The graph passes when `needs` mirrors domain order acyclically, concurrency provides backpressure without cancelling persisted work, and host cancellation is distinguished from build cancellation.

---

# Architectural Invariants

## Invariant GA-271

## Invariant GA-272
Build-critical concurrency groups never cancel in-progress persisted work.

## Invariant GA-273
Host job cancellation implies drain-checkpoint-resumable, never domain build cancellation.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1040 | `needs` expresses ordering only; readiness stays in the orchestrator. |
| ADR-1041 | Concurrency groups provide backpressure and single-active-run guarantees without cancelling persisted work. |
| ADR-1042 | Host cancellation and domain build cancellation are distinct; the former yields a resumable build. |

# Purpose

# Governing Principle

> **One Worker Pool, one runner class, one job responsibility.** Pool separation (submission vs monitoring vs download vs FFmpeg …) is preserved as job separation; pools are never merged onto a single runner.

| Worker Pool (6.3.9) | Job / matrix | Runner class | Held while provider renders? |
| ------------------- | ------------ | ------------ | ---------------------------- |
| Control | `fanout-plan`, orchestration steps | `[self-hosted, control]` | n/a |
| Local Generation | (if used) local-gen job | `[self-hosted, gpu]` | n/a |
| Provider Submission | `render-submit` (matrix) | `[self-hosted, egress]` | **No** — submit-then-release (8.2.6) |
| Provider Monitor | `render-monitor` | `[self-hosted, egress]` | **No** — decoupled poll |
| Download | `render-download` (matrix) | `[self-hosted, egress]` | n/a |
| Validation | validation steps in download job | `[self-hosted, validation]` | n/a |

| FFmpeg | `ffmpeg-compose` | `[self-hosted, ffmpeg]` | n/a |
| Packaging | `package` | `[self-hosted, cpu]` | n/a |
| Metadata | metadata step | `[self-hosted, control]` | n/a |
| Manual Review | `ops.yml` manual-review path | `[self-hosted, control]` | n/a |

2. **FFmpeg isolation.** Composition runs in its own pool so CPU-intensive work never contends with coordination (6.3.9 ADR-888).
3. **No pool merging.** A single runner never plays two pool roles in one job; role bleed reintroduces the coupling the pools exist to prevent.

# Reference Configuration

```yaml

  monitor:    {job: render-monitor,  labels: [self-hosted, egress], hold_during_render: false}
  download:   {job: render-download, labels: [self-hosted, egress]}
  validation: {in: render-download,  labels: [self-hosted, validation]}
  ffmpeg:     {job: ffmpeg-compose,  labels: [self-hosted, ffmpeg]}
  packaging:  {job: package,         labels: [self-hosted, cpu]}
  merge_pools: prohibited
```

* Only submission/monitor/download jobs hold `egress` and provider credentials; control/ffmpeg/packaging jobs do not.
* Manual Review runs under control with no provider credentials.

# Testing Requirements

## Unit Tests
* every 6.3.9 pool maps to exactly one job/runner class;
* submission and monitoring resolve to distinct jobs/leases;
* no job is configured with two pool roles.

## Integration Tests
* submission legs release before provider completion (audited durations), monitoring proceeds independently;
* FFmpeg runs isolated from coordination jobs.

---

# Quality Gate

Pool mapping passes when every 6.3.9 pool maps to one isolated job/runner class, submission and monitoring are separated, FFmpeg is isolated, and no runner plays two pool roles.

# 8.3.5 — Runner Capacity, Concurrency & Autoscaling

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the self-hosted runner fleet is sized, autoscaled, and concurrency-limited so that shot-level fan-out (8.2.6) achieves throughput without exceeding provider limits, fleet capacity, or budget — the host projection of Part 6.1.5 (capacity) and 6.3.16 (backpressure).

---

# Governing Principle

> **Provision for coordination throughput, throttle for provider and budget safety.** Fleet size drives how fast the platform can *coordinate* work; concurrency groups drive how much *provider pressure* it applies.

---

# Capacity Model

| Layer | Control | Bound by |
| ----- | ------- | -------- |
| Matrix legs | `shard_count` (8.2.6) | 256; coordination granularity |
| Simultaneous legs | `max-parallel` | fleet size + provider limit |
| Per-provider inflight | `concurrency` group | provider rate/quotas (6.3.16) |
| Fleet size | autoscaler target | cost + demand |

---

# Autoscaling

Self-hosted pools autoscale (e.g. via an Actions Runner Controller pattern) between a floor and ceiling per pool:

```yaml
autoscale:
  egress:  {min: 2, max: 24, scale_on: queued_submission_jobs}
  ffmpeg:  {min: 0, max: 6,  scale_on: queued_compose_jobs, warm: 1_during_business_hours}
  cpu:     {min: 1, max: 12, scale_on: queued_download_validation}
```

Scaling reacts to queued jobs, not to render wall-clock (submit-then-release means submission jobs are short; the queue signals coordination demand, not provider latency).

---

# Concurrency as Backpressure (→ 6.3.16)

* per-provider `concurrency` caps simultaneous submissions to the provider's safe limit;
* when the cap is reached, additional legs **queue** (delayed), they do not fail;
* backpressure propagates upstream: fan-out honors the cap rather than overwhelming a provider (6.3.16).

---

# Reference Configuration

```yaml
capacity:
  max_matrix_legs: 256
  max_parallel_legs: 8
  per_provider_inflight: {runway: 6, motion: 8, svg: 16, typography: 16}
  autoscale:
    egress: {min: 2, max: 24}
    ffmpeg: {min: 0, max: 6}
    cpu:    {min: 1, max: 12}
  scale_signal: queued_jobs
  on_cap_reached: queue_not_fail
```

---

# Security Requirements

* Autoscaled runners are provisioned ephemeral and capability-labeled; a newly-scaled runner cannot inherit a prior runner's secrets/state.
* Scaling events are audited; the fleet never scales beyond the configured ceiling (cost guard).

---

# Testing Requirements

## Unit Tests
* per-provider inflight never exceeds the configured cap;
* cap-reached queues rather than fails;
* autoscale respects floor/ceiling.

## Integration Tests
* a 100-shot fan-out saturates coordination up to `max-parallel` while holding provider inflight at the cap;
* ffmpeg pool scales from zero to serve a compose job and back down.

## Concurrency Tests
* provider inflight stays ≤ cap under bursty submission;
* backpressure delays, never drops, legs.

---

# Quality Gate

Capacity passes when provider inflight is capped and enforced as queueing backpressure, autoscaling reacts to coordination demand within floor/ceiling, and the fleet never exceeds its cost ceiling.

---

# Architectural Invariants

## Invariant GA-341
Per-provider inflight never exceeds its configured cap; excess work queues, it does not fail.

## Invariant GA-342
Autoscaling reacts to queued coordination jobs, not to provider render wall-clock, and respects floor/ceiling.

## Invariant GA-343
Newly-scaled runners are ephemeral and inherit no prior state or secrets.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1140 | Fleet size governs coordination throughput; concurrency governs provider pressure. |
| ADR-1141 | Concurrency caps act as queueing backpressure (6.3.16), never as failure. |

# 8.3.6 — Runner Lifecycle, Ephemerality & Drain

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the runner's job-scoped lifecycle and its drain behavior on cancellation/eviction, ensuring in-flight work checkpoints to the manifest before a runner disappears — the host projection of Part 6.3.15 (pause/cancel/graceful shutdown).

---

# Governing Principle

> **A runner may vanish at any moment; the manifest must never lose a fact because it did.** Every job is written to survive its runner's sudden death.

---

# Lifecycle

```text
PROVISION → REGISTER(labels) → LEASE(job) → RUN(steps) → PERSIST → DEREGISTER → DESTROY
                                   │
                                   └─ on cancel/timeout/eviction → DRAIN
```

---

# Drain Protocol (→ 6.3.15 / Part 5 drain)

On SIGTERM (cancel, timeout, or scale-in):

```yaml
drain:
  stop_new_work: true            # accept no new task within the job
  checkpoint_inflight: true      # persist current task state to manifest (B-2)
  release_leases: true           # provider submission identity already persisted
  no_task_coupling: true         # draining one leg never fails another
  exit_after_checkpoint: true
```

A drained submission leg has already persisted `provider_task_id` (submit-then-release), so drain loses nothing; a drained download leg checkpoints partial progress and the shot remains reconstructable.

---

# Ephemerality Guarantees

* single-use runner: recycled after one job (GA-121);
* job-scoped workspace: destroyed with the runner; durable data already in manifest/CAS;
* no warm state assumed by the next job (Principle 2).

---

# Reference Configuration

```yaml
lifecycle:
  ephemeral: true
  single_use: true
  drain_signal: SIGTERM
  drain_grace_seconds: 60
  checkpoint_before_exit: true
  on_drain: [stop_new_work, checkpoint_inflight, release_leases, exit]
```

---

# Security Requirements

* On destroy, all secret material and workspace contents are discarded with the ephemeral runner.
* Drain never flushes secrets or payloads to logs/artifacts.

---

# Testing Requirements

## Unit Tests
* SIGTERM triggers checkpoint-before-exit within the grace window;
* draining one leg does not affect siblings.

## Integration Tests
* cancel a build mid-fan-out → all in-flight legs checkpoint; build becomes RESUMABLE with no lost shot;
* scale-in during download → partial progress checkpointed, shot reconstructable.

## Recovery Tests
* kill a runner without grace → the build still resumes from the last persisted manifest state (submit-then-release bounds the loss to unpersisted, safely re-doable work).

---

# Quality Gate

Lifecycle passes when every runner is single-use, drain checkpoints in-flight work before exit without coupling legs, and a build survives arbitrary runner death as RESUMABLE.

---

# Architectural Invariants

## Invariant GA-351
A runner is single-use; the next job assumes no warm state.

## Invariant GA-352
On drain, in-flight work is checkpointed to the manifest before exit, without coupling sibling legs.

## Invariant GA-353
Arbitrary runner death never loses an authoritative fact; the build remains resumable.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1150 | Runners are single-use with a defined drain protocol on SIGTERM. |
| ADR-1151 | Drain checkpoints in-flight work to the manifest before exit; legs stay independent. |
| ADR-1152 | Submit-then-release bounds drain loss to safely-redoable, unpersisted work. |

---

# Acceptance Criteria

Section **8.3.6** is complete when the lifecycle, drain protocol, and ephemerality guarantees are defined, and invariants GA-351…GA-353 with ADRs ADR-1150…ADR-1152 are recorded.

# 8.4 — State, Manifest & Resume on Ephemeral Runners

# 8.4.1 — Purpose & The Stateless-Runner Problem

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section states the central problem Part 8 must solve: the platform is a **stateful, resumable, idempotent** production system (Parts 5–6), yet GitHub Actions runners are **stateless and ephemeral** (8.1.3, 8.3.6). This section frames the problem; 8.4.2–8.4.7 solve it.

---

# Governing Principle

> **The runtime is stateless; the platform is not. Therefore state must live off the runtime and be reattached on every job.** Statefulness is a property of the manifest, never of a runner.

---

# The Problem Precisely

| Domain requirement (Parts 5/6) | Runner reality | Tension |
| ------------------------------ | -------------- | ------- |
| Authoritative, durable build/task state (6.3.4/6.3.5) | Runner filesystem dies with the job | State cannot live on the runner. |
| Resume from any interruption (6.3.12/6.4) | No memory across jobs/runs | Resume must be reconstructable externally. |
| Idempotent, no double provider submission (6.3.17/5.4.1) | Re-runs restart jobs | Re-runs must be safe. |
| Content-addressed cache/CAS (6.2.4) | Actions cache is best-effort, evictable | Cache cannot be the source of truth. |
| Cross-runner continuation (6.4.1) | Jobs are isolated | Handoff must be manifest-mediated. |

---

# The Solution Shape (previewing 8.4.2–8.4.7)

```text
External Durable Store (authoritative)  ── manifest + CAS ──┐
        ▲ read (B-1)          ▲ persist (B-2)               │
        │                     │                             │
   [ job A ]             [ job B ]      ...            [ re-run ]
   ephemeral             ephemeral                     ephemeral
        │                     │                             │
        └── actions/cache + artifacts (accelerators only) ──┘
```

Authoritative state lives in an external durable store; runners read it, act, persist, and vanish; cache/artifacts merely speed this up; resume is a re-run that reconstructs from the store.

---

# Reference Configuration

```yaml
stateless_runner_problem:
  state_home: external_durable_store
  runner_state: none_authoritative
  cache_role: accelerator_only
  resume: manifest_driven_rerun
```

# 8.4.2 — Manifest Persistence Across Ephemeral Jobs

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the Render/Runtime Manifest (Part 5.4 / 6.3.11 / 6.4) is persisted in an external durable store and accessed by ephemeral jobs under the boundary contract (B-1/B-2), so that the manifest — not any runner — is the single source of truth.

---

# Governing Principle

> **Read the manifest, act, persist before ack.** Every job begins by loading authoritative state and ends by persisting its mutation durably before signaling success.

---

# Architectural Position

```text
External Durable Store (e.g. object storage + metadata index)
    manifest/<build_id>.json  (append-only history + current state)   [5.4]
    cas/<hash>                (content-addressed assets)               [6.2.4]
        ▲                              ▲
        │ load-manifest (B-1)          │ persist-manifest (B-2, CAS write)
   ┌────┴──────────────────────────────┴────┐
   │ ephemeral job (any pool)                │
   └─────────────────────────────────────────┘
```

The store provides: atomic conditional writes (compare-and-set on revision), append-only attempt history, and content digests (Part 5.4 integrity).

---

# Access Contract

| Step | Rule |
| ---- | ---- |
| Load | `load-manifest` reads current state + revision (B-1); a job never assumes state from a prior job. |
| Mutate | mutation applied through the legal-transition guard (Part 5.4.4) in the orchestrator, never ad-hoc. |
| Persist | `persist-manifest` writes atomically with compare-and-set on revision (concurrency-safe). |
| Ack | the job signals success only after persist commits (B-2). |
| Conflict | on CAS-revision conflict, the loser reloads and re-evaluates (Part 5.4 concurrency). |

---

# Write-Before-Ack (B-2) Detail

```text
provider submit accepted
    → persist {provider_task_id, SUBMITTED} atomically   [must commit]
    → only then: job step succeeds / releases
If persist fails: the job fails (no ack); re-run re-attempts;
  idempotency (8.4.5) prevents a duplicate provider effect.
```

The manifest write is the commit point; the GitHub job's success is merely a reflection of it.

---

# Reference Configuration

```yaml
manifest_persistence:
  store: external_durable          # object storage + index
  document: manifest/<build_id>.json
  history: append_only              # 5.4
  write: atomic_compare_and_set     # revision-guarded
  integrity: content_digest         # 5.4
  access: {load: B-1, persist: B-2}
  on_conflict: reload_and_reevaluate
  never_source_of_truth: [runner_fs, job_output, cache, artifact]
```

---

# Security Requirements

* Store credentials are federated via OIDC (8.5.2); no long-lived store key on runners.
* Manifest integrity is verified on load (digest); a tampered manifest is an integrity abort (Part 5.4).
* The store is append-only for history; edits/deletes of attempt records are rejected.

---

# Testing Requirements

## Unit Tests
* a job refuses to act on assumed (non-loaded) state (B-1);
* ack before persist is impossible (B-2);
* an illegal manifest transition is rejected by the guard (5.4.4);
* a revision conflict causes reload-and-reevaluate, not a lost update.

## Integration Tests
* two jobs mutating different entries commit without lost updates;
* two jobs mutating the same entry: CAS serializes, loser re-evaluates;
* crash after persist, before ack → replay is a no-op.

## Failure Injection Tests
* tampered manifest digest → integrity abort on load;
* store write failure → job fails without ack; re-run recovers.

---

# Quality Gate

Manifest persistence passes when the manifest in the external store is the sole source of truth, all access obeys B-1/B-2 with atomic revision-guarded writes and append-only history, and no runner-local artifact is ever authoritative.

---

# Architectural Invariants

## Invariant GA-411
The manifest in the external durable store is the single source of truth; runner FS, job outputs, cache, and artifacts are never authoritative.

## Invariant GA-412
Every job obeys load-then-act (B-1) and persist-before-ack (B-2) with atomic, revision-guarded writes.

## Invariant GA-413
Manifest history is append-only and digest-verified; tampering is an integrity abort.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1210 | The manifest lives in an external durable store and is the single source of truth. |
| ADR-1211 | Manifest writes are atomic, revision-guarded, and append-only for history. |
| ADR-1212 | Manifest access obeys B-1/B-2; the GitHub job's success reflects the manifest commit, not vice versa. |

---

# Acceptance Criteria

Section **8.4.2** is complete when the external store, access contract, write-before-ack detail, and integrity model are defined, and invariants GA-411…GA-413 with ADRs ADR-1210…ADR-1212 are recorded.

# 8.4.4 — Checkpoint & Resume Across Re-runs

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the platform resumes an interrupted build by re-running, reconstructing all needed work from the manifest, and generating an immutable Recovery Plan — the host projection of Part 6.3.12 (Checkpoint & Resume) and Part 6.4 (Manifest & Resume).

---

# Governing Principle

> **Resume by re-run; reconstruct from the manifest; never repair by hand.** A re-run is always safe because idempotency (8.4.5) makes redone work harmless.

---

# Resume Flow

```text
trigger re-run (ops.yml or GitHub re-run failed jobs) with build_id
    → load manifest (authoritative state)                    [B-1]
    → compute remaining work:
         not_ready = { shot | state ∉ {READY,CACHED,COMPOSED} }
         in_flight = { shot | SUBMITTED/RUNNING with provider task }
    → generate immutable Recovery Plan                        [6.3.12]
         reattach in_flight (no re-submit); re-fan-out not_ready
    → resume execution deterministically (8.2.6 planner regenerates identical matrix)
```

Resume outcomes (Part 6.4): resumed-complete, resumed-partial-then-held, or denied (recovery contract unmet).

---

# Recovery Plan

```json
{
  "build_id": "20260720-001",
  "generation": 2,
  "reattach": ["sh-014 (runway task t-abc)", "..."],
  "refanout": ["sh-031", "sh-032"],
  "already_ready": 40,
  "recovery_contract_satisfied": true
}
```

The Recovery Plan is immutable per resume attempt and auditable, even when recovery is denied (Part 6.4).

---

# Determinism Requirement

Because the fan-out planner is deterministic (GA-263), a re-run regenerates the **identical** matrix for the remaining set, so resume is reproducible and never reorders or duplicates work.

---

# Reference Configuration

```yaml
resume:
  trigger: [ops_resume, github_rerun_failed_jobs]
  reconstruct_from: manifest
  reattach_inflight: true            # no re-submission
  refanout_not_ready: true
  recovery_plan: immutable_per_attempt
  outcomes: [complete, partial_then_held, denied]
  deterministic_matrix: required
```

---

# Security Requirements

* Resume never re-materializes provider submissions for already-submitted tasks (reattach), preventing double spend.
* A denied recovery (contract unmet) is recorded and does not silently proceed.

---

# Testing Requirements

## Unit Tests
* remaining-work computation excludes READY/CACHED/COMPOSED and reattaches in-flight;
* the resume matrix equals the original matrix for the remaining set (determinism).

## Integration Tests
* interrupt after submission → resume reattaches in-flight tasks, re-fans-out only not-ready, completes with no double submission;
* interrupt before composition → resume reaches the barrier and composes once.

## Recovery Tests
* multiple sequential interruptions → each resume advances monotonically; no work is lost or duplicated;
* denied recovery (corrupt/incomplete contract) is recorded and halts, not proceeds.

---

# Quality Gate

Resume passes when a re-run reconstructs remaining work from the manifest deterministically, reattaches in-flight tasks without re-submission, produces an immutable auditable Recovery Plan, and converges to completion or a recorded hold.

---

# Architectural Invariants

## Invariant GA-431
Resume is a manifest-driven re-run that reconstructs remaining work deterministically.

## Invariant GA-432
In-flight provider tasks are reattached on resume, never re-submitted.

## Invariant GA-433
Every resume attempt produces an immutable, auditable Recovery Plan; denied recoveries are recorded, not silently continued.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1230 | Resume is a deterministic, manifest-driven re-run, never manual repair. |
| ADR-1231 | In-flight tasks are reattached; resume never re-submits an existing provider task. |
| ADR-1232 | Each resume yields an immutable Recovery Plan; denials are auditable. |

---

# Acceptance Criteria

Section **8.4.4** is complete when the resume flow, Recovery Plan, and determinism requirement are defined, and invariants GA-431…GA-433 with ADRs ADR-1230…ADR-1232 are recorded.

# 8.4.5 — Idempotency & Duplicate Prevention on Re-runs

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines how the platform guarantees that re-runs, retried jobs, redelivered events, and lost descriptors never cause a duplicate externally-visible effect — especially a double provider submission or a double publish — the host projection of Part 6.3.17 and Part 5.4.1.

---

# Governing Principle

> **Every externally-visible action is idempotent, keyed on content, and admitted at most once.** Re-running is safe by construction, not by luck.

---

# Idempotency Keys

| Action | Idempotency key | Store of record |
| ------ | --------------- | --------------- |
| Provider submission | render cache key (5.6.3) | manifest provider-task record |
| CAS write | content hash (6.2.4) | CAS (write is content-addressed, naturally idempotent) |
| Composition | build_id + asset-set hash | manifest COMPOSED state |
| Publish | build_id + package hash | manifest PUBLISHED + publish target |

---

# Idempotent Admission (Provider Submission)

```text
before submit(shot):
  key = render_cache_key(shot)                       # 5.6.3
  rec = manifest.provider_task(key)
  if rec exists and rec.state ∈ {SUBMITTED,RUNNING,DOWNLOADING,READY}:
      SKIP (no re-submission)                         # duplicate prevented
  else:
      submit; persist {key, provider_task_id} (B-2)
```

Host identifiers (`run_attempt`, job re-run) never key idempotency; only the content-derived cache key does (GA-103).

---

# Redelivery & Lost-Descriptor Handling

* **Redelivered event** (duplicate `repository_dispatch`): idempotent trigger (8.2.2) → no second build.
* **Retried job**: reloads manifest, sees existing provider task → SKIP.
* **Lost matrix descriptor**: planner regenerates identical matrix (GA-263); admission still keyed on cache key → no duplicate.
* **Provider accepted but persist lost**: monitor reconciles the orphan provider task against the manifest (Part 5.4) rather than creating a new one.

---

# Reference Configuration

```yaml
idempotency:
  provider_submit_key: render_cache_key
  cas_write: content_addressed
  compose_key: build_id+assetset_hash
  publish_key: build_id+package_hash
  host_ids_as_keys: prohibited          # GA-103
  on_existing: skip
  orphan_provider_task: reconcile_not_duplicate
```

---

# Security Requirements

* Idempotency prevents a compromised re-run from double-charging providers or double-publishing.
* Reconciliation of orphaned provider tasks is authenticated and manifest-verified, never inferred from unauthenticated provider chatter.

---

# Testing Requirements

## Unit Tests
* re-submission for an existing cache-key task is a SKIP;
* host identifiers are never used as idempotency keys;
* CAS writes are content-addressed and repeatable.

## Integration Tests
* re-run a completed build → zero new provider submissions, zero new publishes;
* retried submission job → SKIP on already-submitted shots.

## Failure Injection Tests
* provider accepted but persist lost → monitor reconciles the orphan, does not duplicate;
* duplicate publish attempt → no-op (idempotent publish).

---

# Quality Gate

Idempotency passes when every externally-visible action is admitted at most once keyed on content, re-runs produce no duplicate effects, and orphaned provider tasks are reconciled rather than duplicated.

---

# Architectural Invariants

## Invariant GA-441
Every externally-visible action is idempotent and keyed on content, never on host identifiers.

## Invariant GA-442
A re-run or retried job produces no duplicate provider submission, CAS write, composition, or publish.

## Invariant GA-443
Orphaned provider tasks are reconciled against the manifest, never duplicated or fabricated.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1240 | Idempotency keys are content-derived; host identifiers are barred as keys. |
| ADR-1241 | Admission SKIPs any action whose content key already exists in a non-terminal/terminal success state. |
| ADR-1242 | Orphaned provider tasks are reconciled, guaranteeing at-most-once external effect. |

---

# Acceptance Criteria

Section **8.4.5** is complete when idempotency keys, idempotent admission, redelivery/lost-descriptor handling, and reconciliation are defined, and invariants GA-441…GA-443 with ADRs ADR-1240…ADR-1242 are recorded.

> **Prefer a short-lived token minted at need over a long-lived secret stored at rest.** Where OIDC federation exists, use it; where it does not, scope the static secret to an environment and rotate it.

---

# Secret Scoping Hierarchy

| Scope | Use | Reachable by |
| ----- | --- | ------------ |
| Environment secret (e.g. `production-execute`) | production provider/store creds | only jobs targeting that environment, post-approval (8.2.4) |
| Repository secret | shared non-production config | any job (avoid for high-value creds) |
| OIDC federation | store & cloud access | jobs with `id-token: write`, exchanged at runtime |

---

# OIDC Flow

```text
job (id-token: write)
   → request OIDC token from GitHub (claims: repo, ref, environment)
   → exchange at cloud/store/provider for a short-lived credential
   → use within the step; token expires shortly after
No long-lived key is stored or placed on the runner.
```

The trust policy on the cloud/store side restricts which repo/ref/environment may assume which role (defense in depth).

---

# Reference Configuration

```yaml
secrets:
  prefer: oidc
  environment_scoped: [production-execute, publish]
  repository_secrets: non_production_only
  oidc:
    id_token: write_where_needed
    claims: [repo, ref, environment]
    trust_policy: restrict_by_repo_ref_environment
  rotation:
    static_secrets: mandatory_periodic
```

---

# Security Requirements

* OIDC trust policies are scoped to the specific repo/ref/environment; a fork or other repo cannot assume production roles.
* Any remaining static secret is environment-scoped and on a rotation schedule.
* `id-token: write` is granted only to jobs that federate; others do not request it.

---

# Testing Requirements

## Unit Tests
* only environment-targeting jobs can read environment secrets;
* jobs not needing federation do not request `id-token: write`.

## Integration Tests
* an OIDC exchange yields a short-lived credential scoped by claims;
* a run from a non-approved ref/environment is denied the production role by the trust policy.

## Failure Injection Tests
* attempt to assume a production role from a fork/PR context → denied by trust policy.

---

# Quality Gate

Secret management passes when OIDC is used wherever supported, static secrets are environment-scoped and rotated, trust policies restrict role assumption by repo/ref/environment, and `id-token` is minimally granted.

---

# Architectural Invariants

## Invariant GA-511
OIDC-federated short-lived credentials are used wherever supported; long-lived secrets are minimized.

## Invariant GA-512
Static secrets are environment-scoped and rotated; production secrets are reachable only post-approval.

## Invariant GA-513
OIDC trust policies restrict role assumption by repo/ref/environment, excluding forks and other repos.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1310 | OIDC federation is preferred over stored secrets for store/provider/cloud access. |
| ADR-1311 | Static secrets are environment-scoped and rotated. |

# 8.6.2 — Cache Strategy over CAS

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the caching strategy that layers `actions/cache` over the authoritative CAS (Part 6.2.4), with content-derived keys and incremental reuse (Part 6.2.6), so that unchanged work is never recomputed or re-rendered.

---

# Governing Principle

> **Cache keys are content hashes; a hit is trusted only after it matches the expected CAS hash.** The cache accelerates CAS; it never contradicts it.

---

# Key Derivation

```text
cache_key(x)   = "v2-" + cas_hash(inputs_of(x))     # deterministic, content-addressed
restore_keys   = ["v2-" + prefix(cas_hash), "v2-"]   # graceful fallback
on hit: verify restored content hash == cas_hash(inputs_of(x)); else discard + fetch CAS
```

Keys cover: planning-stage outputs (story/screenplay/shot/storyboard), dependency toolchains, and render assets keyed by render cache key (5.6.3).

---

# Incremental Reuse (→ 6.2.6)

* a shot whose render cache key is already satisfied (CAS hit) is excluded from fan-out (8.2.6), incurring zero provider spend;
* planning stages with unchanged inputs are restored from cache, making re-planning near-instant;
* invalidation follows CAS content change (6.2.6), never a time-based expiry.

---

# Reference Configuration

```yaml
cache_strategy:
  key: cas_hash_derived
  version_prefix: v2
  restore_keys: prefix_fallback
  validate_on_hit: true
  authoritative_fallback: cas
  invalidation: content_change_only
  covers: [planning_stages, toolchains, render_assets]
```

---

# Security Requirements

* a cache hit failing hash validation is discarded (cache poisoning defense, 8.4.3);
* no secret is ever part of a cache key or cache content.

---

# Testing Requirements

## Unit Tests
* identical inputs → identical cache key; changed inputs → new key (content-addressed);
* a hit with mismatched content hash is discarded and CAS is fetched.

## Integration Tests
* unchanged article → planning fully cache-served;
* unchanged shots → excluded from fan-out with zero provider spend;
* cache disabled → still correct via CAS, slower.

---

# Quality Gate

Cache strategy passes when keys are content-derived, hits are hash-validated against CAS, invalidation is content-based, and incremental reuse demonstrably eliminates redundant planning and rendering cost.

---

# Architectural Invariants

## Invariant GA-611
Cache keys are content-derived; invalidation is by content change, never time.

## Invariant GA-612
A cache hit is trusted only after matching the expected CAS hash; otherwise it is discarded.

## Invariant GA-613
Incremental reuse excludes unchanged shots from fan-out, incurring zero provider spend.

---

# Architectural Decision Records

| ADR | Decision |
| --- | -------- |
| ADR-1410 | Cache is layered over CAS with content-derived keys and content-based invalidation. |
| ADR-1411 | Cache hits are hash-validated before trust; CAS is the fallback authority. |
| ADR-1412 | Incremental reuse is the mechanism eliminating redundant provider spend. |

---

# Acceptance Criteria

# 8.8.2 — Retry, Backoff & Failure Classification

# Purpose

This section defines how failures are retried and classified on GitHub Actions, ensuring the orchestrator's classification (Part 6.3.13 / 5.4.5) governs — not GitHub's blunt job-level auto-retry.

---

# Governing Principle

> **Retry is a domain decision, not a host reflex.** GitHub can restart a job, but whether, how, and how often to retry a *task* is classified by the orchestrator.

---

# Two Retry Layers

| Layer | Mechanism | Governs |
| ----- | --------- | ------- |
| Task retry | orchestrator classification (6.3.13) | transient vs terminal; attempt count; backoff; Manual Review escalation |
| Job retry | GitHub re-run / bounded step retry | infrastructure hiccups only, subordinate to task classification |

GitHub's automatic job retry is disabled or neutralized so it cannot cause a second provider submission; idempotency (8.4.5) guarantees safety even if a job is re-run.

---

# Classification (→ 6.3.13 / 5.4.5)

```text
on task failure:
  class = classify(error)              # transient | rate_limited | terminal | provider_not_found | timeout
  if class ∈ retryable and attempt < max_retry:
       schedule retry with backoff (delayed visibility)   # queue, not hold
  elif attempt >= max_retry or class terminal:
       route to Manual Review (Part 5.4.5)
```

Backoff is realized as delayed re-queue (monitor/next run), never as a runner sleep (Principle 4).

---

# Reference Configuration

```yaml
retry:
  task_layer: orchestrator_classification
  job_layer: subordinate_infra_only
  github_auto_retry: neutralized
  classes: [transient, rate_limited, terminal, provider_not_found, timeout]
  max_retry: per_profile
  backoff: delayed_requeue_not_sleep
  exhausted: manual_review
  idempotent_on_rerun: true
```

---

# Security Requirements

* a re-run/retry never double-charges providers (idempotency, GA-441);
* classification reasons are logged (secret-free) and surfaced as annotations (8.7.2).

---

# Testing Requirements

* GitHub auto-retry cannot trigger a second submission.

* a transient provider error retries with backoff and succeeds;

* backoff is delayed re-queue, not a runner sleep.

## Failure Injection Tests
* rate-limit → backoff/retry within provider caps;
* terminal error → immediate Manual Review, no retry.

---

# Quality Gate

Retry passes when task classification governs, GitHub auto-retry cannot cause duplicate effects, backoff is delayed re-queue, and exhaustion routes to Manual Review.

---

## Invariant GA-811

## Invariant GA-812

---

# Governing Principle

> **A timeout or cancellation stops a runner, not a build.** Every interruption drains and checkpoints; the build becomes resumable, never corrupted.

---

# Timeout Policy

| Job | `timeout-minutes` rationale |
| --- | --------------------------- |
| plan | generous ceiling over the 180s target (8.2.3) |
| render-submit | short (submit+persist only, 8.2.6) |

* host cancellation (manual/timeout/superseded) → drain + checkpoint → RESUMABLE;
* domain cancellation (ops.yml) → orchestrator transitions to CANCELLED with compensation (6.3.14);
* the two are never conflated (GA-273).

---

# Reference Configuration

```yaml
shutdown:
  timeouts:
    submit: 15
    monitor: 30
    download: 20
    ffmpeg: sized_to_worst_case
  on_timeout: drain_checkpoint_resumable
  host_cancel: resumable
  domain_cancel: cancelled_with_compensation
  grace_seconds: 60
```

---

# Security Requirements

* drain never flushes secrets/payloads to logs or artifacts;
* a superseded run's cancellation cannot corrupt the manifest (atomic writes, 8.4.2).

---

# Testing Requirements

## Unit Tests
* timeout triggers checkpoint-before-exit within grace;
* host vs domain cancellation resolve to resumable vs cancelled-with-compensation.

## Integration Tests
* timeout mid-fan-out → build RESUMABLE, resume completes (8.4.4);
* domain cancel → CANCELLED with compensation, no orphaned provider spend.

## Failure Injection Tests
* ungraceful kill (no drain) → build still resumes from last persisted state.

# Purpose

---

# Governing Principle

> **Every operational scenario has one documented, safe, idempotent procedure.** Operators follow runbooks that invoke verbs; they never improvise against the manifest.

---

# Runbooks

## RB-1 — Resume a Stuck/Interrupted Build

```text
1. ops.yml → action=resume, build_id=…
2. orchestrator loads manifest, generates Recovery Plan (8.4.4)
3. reattach in-flight, re-fan-out not-ready
4. verify convergence to COMPLETED or documented hold
```

## RB-2 — Re-run Failed Shots

```text
1. GitHub "re-run failed jobs" OR ops.yml action=retry-failed
2. idempotency ensures no double submission (8.4.5)
3. classification governs retry vs Manual Review (8.8.2)
```

## RB-3 — Force Manual Review

```text
1. ops.yml → action=manual-review, build_id, shot(s)
2. orchestrator routes shot(s) to Manual Review; build holds at barrier
3. no partial trailer composed
```

## RB-4 — Emergency Stop

```text
1. ops.yml → action=emergency-stop, build_id (elevated auth)
2. orchestrator drains, cancels with compensation (6.3.14)
3. in-flight provider tasks recorded; spend bounded; build CANCELLED
```

---

# Reference Configuration

```yaml
runbooks:
  RB1_resume: {surface: ops.yml, verb: resume, idempotent: true}
  RB2_retry:  {surface: [github_rerun, ops.yml], verb: retry_failed, idempotent: true}
  RB3_manual_review: {surface: ops.yml, verb: manual_review}
  RB4_emergency_stop: {surface: ops.yml, verb: emergency_stop, auth: elevated}
  all_actions: auditable_in_manifest
```

# 8.8.6 — Architectural Invariants (Part 8)

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section consolidates the invariants that Part 8 guarantees as a whole. Each subsection above carries its own local GA-series invariants; this section (a) states the small set of **top-level Part 8 invariants** that summarize them, and (b) indexes the local invariant families by section for traceability.

---

# Governing Principle

> **If GitHub Actions vanished tomorrow and were replaced by another host, every one of these invariants would still have to hold — because they are properties of the platform, merely enforced here on this host.**

---

# Top-Level Part 8 Invariants

## Invariant P8-1 — Host Holds No Truth
No authoritative domain state exists only on GitHub Actions; the entire build is reconstructable from the external manifest/CAS. *(summarizes GA-101, GA-401, GA-411, GA-451)*

## Invariant P8-2 — Scheduling, Not Policy
Workflow YAML expresses scheduling only; all domain policy lives in the host-agnostic orchestrator. *(GA-102, GA-201, GA-731)*

## Invariant P8-3 — Submit, Never Hold
No runner is held for external provider render wall-clock; submission persists identity and releases. *(GA-262, GA-332, GA-813)*

## Invariant P8-4 — At-Most-Once External Effect
Re-runs, retries, redeliveries, and lost descriptors never cause a duplicate provider submission, CAS write, composition, or publish. *(GA-264, GA-441, GA-442, GA-812)*

## Invariant P8-5 — Resume by Manifest, Not by Host
Resume is a deterministic, manifest-driven re-run correlated by `build_id`; host identifiers are never authoritative. *(GA-103, GA-403, GA-431, GA-451)*

## Invariant P8-6 — Fail Closed
Ambiguity — missing input, invalid config, unverified approval, budget breach, integrity mismatch, capability shortfall — halts the affected work rather than proceeding. *(GA-105, GA-222, GA-231, GA-321, GA-361, GA-631)*

## Invariant P8-7 — Quality and Cost Are Gated
High-cost work is budget- and approval-gated before fan-out; composition/publication are blocked while any Quality Gate fails. *(GA-231, GA-631, GA-731)*

## Invariant P8-8 — Least Privilege, Pinned, Isolated
Secrets are step-scoped and OIDC-federated where possible; permissions are least-privilege; third-party actions are SHA-pinned; forks get no secrets. *(GA-501, GA-511, GA-531, GA-541, GA-551)*

## Invariant P8-9 — Single-Crossing, No Partial Trailer
Composition happens at most once per build behind an all-or-hold barrier; no partial or unverified trailer is ever produced. *(GA-251, GA-268, GA-731)*

## Invariant P8-10 — Host-Agnostic Domain
Nothing in Part 8 precludes running the same domain locally or on a future distributed host; the Actions binding is one of many. *(GA-105, GA-142)*
