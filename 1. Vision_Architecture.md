# 1.0.3 Platform v3.4.0 Amendment — GPT-5.6 Pre-Render Storyboard Preview

## Purpose and precedence

This amendment defines a **zero-Runway previsualization path** for inspecting the complete creative plan before any Runway credit can be consumed. It extends the planning/render separation and human-approval rules. Where the existing Preview Rendering Profile in Part 5 calls `runway_preview`, this amendment governs the earlier pre-Runway gate: a Runway-produced preview cannot satisfy a gate whose purpose is to prevent the first Runway call.

The feature is implemented behind the explicit `off | storyboard_animatic` mode. The production default remains `off`; enabling it requires the Responses API adapter, bounded budget, protected workflow, and human approval. Non-billable acceptance is part of the repository; bounded live OpenAI evaluation remains an operational release check.

## Capability boundary

| Capability | GPT-5.6 previsualization responsibility | Boundary |
| --- | --- | --- |
| Scene decomposition, dramatic direction, camera intent, narration, tempo, Storyboard, Shot List, Image Prompt, Video Prompt | Generate typed planning proposals from the sealed Insight-derived Story and Screenplay; deterministic validators and a human remain authoritative. | No provider call is authorized by model output. |
| Storyboard frame generation | GPT-5.6 may invoke the Responses API `image_generation` tool; the pixels are generated by the selected GPT Image model and its resolved model/version shall be recorded. | This consumes OpenAI tokens/image charges, but consumes zero Runway credits. It is not described as GPT-5.6 directly emitting image bytes. |
| Animatic MP4 composition | FFmpeg assembles approved-duration still frames, narration captions or an explicitly selected preview voice track, and timing metadata. | FFmpeg does not synthesize shot motion or replace a video-generation model. |
| Final motion-video generation | Outside GPT-5.6. Runway remains the configured production video provider unless a separately approved `VideoRenderProvider` is introduced. | GPT-5.6 shall never be represented as directly converting a Video Prompt into MP4. OpenAI Sora, if ever adopted, is a separate video provider and requires its own ADR, budget, approval, and adapter. |

Normative OpenAI capability references are [Using GPT-5.6](https://developers.openai.com/api/docs/guides/model-guidance?model=gpt-5.6), [Image generation](https://developers.openai.com/api/docs/guides/image-generation), and [Video generation with Sora](https://developers.openai.com/api/docs/guides/video-generation). These references establish the model/tool/provider separation; repository policy and the contracts below remain authoritative for platform admission and approval.

## Canonical flow

```text
sealed Insight Article
  -> approved Story
  -> approved Screenplay (the canonical Script)
  -> GPT-5.6 previsualization proposal
  -> deterministic Scene / Shot / Storyboard / Prompt validation
  -> GPT Image storyboard frames through the image_generation tool
  -> FFmpeg storyboard animatic + review sheet
  -> Storyboard Preview Quality Gate
  -> human storyboard-preview-approval
  -> render-approval and Runway execution
  -> final MP4
```

The operator-facing review order may present Storyboard before Shot List, but the canonical data dependency remains Screenplay -> Shot List -> Storyboard so that every panel resolves to exactly one `shot_id`. `Video Prompt` artifacts are generated and sealed for inspection but shall not be submitted to Runway before both the preview approval and the existing render approval are valid.

## Operating mode

```yaml
pre_render_preview:
  mode: off | storyboard_animatic
  planning_model: gpt-5.6-sol
  image_generation: responses_image_generation_tool
  compositor: ffmpeg
  narration_preview: captions | offline_tts | approved_tts
  runway_contact_before_preview_approval: forbidden
```

`off` preserves legacy and local compatibility while this amendment is draft. A production profile that enforces this amendment shall require `storyboard_animatic`; callers cannot downgrade it to `off`. The requested model, resolved GPT-5.6 model, reasoning effort, prompt version, structured-output contract, image tool/model provenance, and FFmpeg version shall be recorded.

## Review and approval contract

The review surface shall expose, for every scene and shot:

* scene purpose, dramatic beat, direction, and visible action;
* shot size, lens intent, angle, camera position and intended movement;
* narration text, timecode, silence, and source/assumption lineage;
* shot duration, transition, cumulative runtime, and tempo/rhythm classification;
* Storyboard frame, Image Prompt, and sealed Video Prompt;
* deterministic findings and GPT-5.6 quality-evaluation findings, clearly distinguished.

GPT-5.6 quality evaluation is advisory evidence. Deterministic schema, duration, referential-integrity, security, prompt-lineage, and budget checks remain authoritative. Human approval binds the exact Insight, Story, Screenplay, Shot List, Storyboard, Camera Plan, narration/timing plan, Image Prompt set, Video Prompt set, generated frames, animatic manifest, and quality-report hashes. Changing any bound member invalidates approval and keeps Runway unreachable.

The animatic is non-publishable and must carry a visible preview marker. It can validate intended composition, shot order, narration, and tempo. It cannot prove generated motion fidelity, actor performance, lip synchronization, temporal consistency inside a shot, or final photorealism; those limitations shall be visible on the review surface.

## Cost and security boundary

Before any OpenAI image call, the platform shall estimate and enforce a separate planning-preview budget, maximize exact-key cache reuse, and fail closed when price or usage cannot be bounded. OpenAI preview cost shall never be accounted as Runway credits or Runway attempts. `RUNWAY_API_KEY` is unavailable to every previsualization job. `OPENAI_API_KEY` is unavailable to Runway and FFmpeg-only jobs. Prompts, credentials, unpublished source bodies, and generated frames shall not be written to public logs.

## ADRs

| ADR | Decision |
| --- | --- |
| ADR-1.0.3-A | A zero-Runway storyboard animatic is the mandatory review artifact before the first Runway call in enforcing production profiles. |
| ADR-1.0.3-B | GPT-5.6 owns structured previsualization judgment; GPT Image owns generated pixels; FFmpeg owns deterministic animatic composition. |
| ADR-1.0.3-C | Canonical data dependency remains Shot List before Storyboard even when the UI presents Storyboard first. |
| ADR-1.0.3-D | GPT-5.6 cannot serve as a Video Prompt-to-MP4 provider; any Sora adoption is a separate provider decision. |
| ADR-1.0.3-E | Preview approval binds all creative, prompt, frame, timing, and quality evidence and is invalidated by any change. |

## Acceptance criteria

The amendment is complete when an enforcing production run can generate and review scene construction, direction, camera work, narration, tempo, Storyboard, Shot List, Image Prompts, Video Prompts, still frames, and a non-publishable FFmpeg animatic with zero Runway calls; OpenAI image cost is bounded and separately measured; GPT-5.6/image/FFmpeg provenance is explicit; the review limitations are disclosed; any bound change invalidates approval; and both preview approval and render approval are required before Runway submission.

# 1.0.2 Platform v3.3.0 Amendment — Persona Council

## Purpose and precedence

Platform v3.3.0 introduces **Persona Council**, a bounded multi-agent deliberation stage implemented with the OpenAI Agents SDK. This amendment is normative for v3.3.0. It extends, but does not weaken, the v2.1 Agent Review, deterministic generation, provider isolation, human approval, immutability, and publication contracts. Where this amendment conflicts with a v2.1 statement that multi-agent behavior is out of scope, this amendment governs only the Persona Council boundary. Agent Review remains a separate, single-agent, read-only boundary.

Persona Council creates one evidence-backed protagonist persona before Story generation. It is not a free-form agent conversation, autonomous writers' room, renderer, approver, or publisher.

## Operating mode and compatibility

```yaml
persona_deliberation:
  mode: off | council
```

| Mode | Required behavior |
| --- | --- |
| `off` | Preserve the Platform v3.2.0 Article-to-Story path. No Persona Council call, artifact, credential, or approval is required. |
| `council` | Run the bounded Persona Council for the exact deliberation key, validate and seal its artifacts, and require Persona Approval before Story generation. |

The default remains `off` until the v3.3.0 implementation, evaluation suite, schemas, and protected approval workflow are released together. A build shall record the effective mode.

## Architectural position

```text
sealed Article + explicit Creative Brief
  -> Persona Council preflight
  -> Audience Researcher / Empathy and Narrative Analyst / Brand Strategist
  -> Red-Team Critic
  -> Persona Manager synthesis
  -> deterministic Persona Quality Gate
  -> human Persona Approval
  -> deterministic Story Engine
  -> deterministic Screenplay Engine
  -> deterministic Shot Planner and Storyboard Engine
  -> deterministic Quality Gates
  -> Agent Review Boundary (optional review mode)
  -> human render-approval
  -> Rendering Platform
  -> human publication-approval
  -> Publication
```

Direct Article-to-Video conversion remains prohibited. Persona Council is an upstream narrative-design control and cannot skip Story, Screenplay, Shot, Storyboard, review, rendering, or approval stages.

## Orchestration and ownership

Persona Council shall use the manager-owned **agents-as-tools** pattern. The Persona Manager owns the only canonical final persona and may invoke four narrowly scoped specialists as bounded agent tools. Handoffs are prohibited because no specialist may take ownership of the workflow or final artifact.

| Agent | Authority and output |
| --- | --- |
| Audience Researcher | Extract audience context, job-to-be-done, decision pressure, vocabulary, and source evidence. |
| Empathy and Narrative Analyst | Propose desire, fear, contradiction, human stakes, and a plausible 30-second emotional movement. |
| Brand Strategist | Test alignment with Insynergy thought leadership, Decision Design, executive relevance, and tone. |
| Red-Team Critic | Identify unsupported biography, stereotypes, borrowed stakes, manipulation, contradictions, and weak causality; it cannot rewrite the persona. |
| Persona Manager | Synthesize exactly one canonical persona, resolve or expose objections, and label every non-source assertion as an assumption. |

Specialists shall not call one another. They exchange only schema-valid artifacts through the orchestrator. The finite protocol is:

1. Three proposal specialists each run exactly once against the same sealed inputs.
2. The Red-Team Critic runs exactly once against the sealed proposal set.
3. The Persona Manager runs exactly once to synthesize one canonical persona and an objection-resolution record.
4. Deterministic validators evaluate schemas, evidence, cardinality, provenance, policy, and hashes.
5. The build enters `AWAITING_PERSONA_APPROVAL`; no Story generation is permitted before an attributable human approval.

The normative deliberation limit is one proposal round, one critique pass, and one synthesis pass. Automatic recursive debate, majority voting, unbounded turns, hidden negotiation, and self-approval are prohibited.

## Canonical artifacts

Persona Council shall seal the following immutable artifacts:

* `persona-proposals.json`;
* `persona-red-team-report.json`;
* `persona-deliberation.json`;
* `persona.json`;
* `persona-quality-report.json`;
* `persona-approval-binding.json` after approval.

Artifacts contain structured conclusions, evidence references, declared assumptions, objections, and resolutions. They shall not contain chain-of-thought, hidden reasoning, credentials, raw environment state, or unrestricted conversation transcripts.

The canonical persona shall contain exactly one protagonist role, one job-to-be-done, one dominant desire, one dominant fear, one internal contradiction, one decision pressure, one authority boundary, and one emotional-arc candidate. Demographic facts, biography, quotations, medical conditions, vulnerable settings, and personal history may appear only when supplied by the source Article or explicit Creative Brief; otherwise they are prohibited rather than inferred.

## Authority and approval

1. Source fidelity, schema validation, deterministic policy, and Persona Quality Gate results are authoritative over model output.
2. Persona Council may propose and synthesize but may not approve, generate Story artifacts, invoke providers, or mutate previously sealed artifacts.
3. A human approves the exact tuple of Article hash, Creative Brief hash, canonical persona hash, quality-report hash, deliberation contract version, and policy version.
4. Any changed member of that tuple invalidates Persona Approval and all unapproved downstream derivations.
5. `PASS` means eligible for human review, never automatically approved.
6. `MANUAL_REVIEW_REQUIRED`, `UNAVAILABLE`, or `ERROR` fails closed before Story generation.

## Security, privacy, and observability

The Article and Creative Brief are untrusted model data. Embedded instructions, requests for tools or secrets, output-schema changes, and role changes shall be treated as content rather than instructions. Persona specialists have no filesystem, network, MCP, rendering, approval, publication, or external-write tools. Only manager-owned specialist tools are allowed.

The SDK may emit traces under an approved policy, but model-visible source bodies, unpublished creative material, personal data, and raw outputs shall be excluded from remote trace payloads by default. The platform source of truth is the content-addressed artifact set and append-only events, not SDK session state or trace storage.

## ADRs

| ADR | Decision |
| --- | --- |
| ADR-1.0.2-A | Persona Council is an upstream, bounded narrative-design stage owned by the Build Orchestrator. |
| ADR-1.0.2-B | The Persona Manager uses specialists as tools and retains final artifact ownership; handoffs are prohibited. |
| ADR-1.0.2-C | Deliberation has one proposal round, one Red-Team pass, and one synthesis pass. |
| ADR-1.0.2-D | Structured conclusions are retained; chain-of-thought and free-form transcripts are prohibited artifacts. |
| ADR-1.0.2-E | Human Persona Approval is mandatory before Story generation in `council` mode. |
| ADR-1.0.2-F | Agent Review remains a separate single-agent, read-only post-planning reviewer. |

## Acceptance criteria

This amendment is complete only when the same sealed input and configuration resolve to one immutable selected artifact set per deliberation key; all specialists obey their typed and tool-free contracts; no unbounded debate or handoff is possible; unsupported demographic or vulnerable-setting invention is blocked; every persona assertion is evidence-backed or explicitly declared as an assumption; a human approval binds the exact persona evidence set; and no Story, render, approval, or publication side effect is reachable from Persona Council.

# 1.0.1 Specification v2.1 Amendment — Agent Review Mode

## Purpose and precedence

Specification v2.1 introduces **Agent Review Mode** as a planning-layer review boundary implemented with the OpenAI Agents SDK. This section is normative. Where a v2.0-era statement conflicts with this amendment, this amendment governs. All existing deterministic generation, validation, rendering, approval, publication, and immutability requirements remain in force unless this section explicitly refines them.

Agent Review Mode adds review evidence; it does not add an autonomous author, director, renderer, approver, or publisher.

## Operating modes

The platform shall expose one explicit configuration value:

```yaml
agent_review:
  mode: off | review
```

| Mode | Required behavior |
| --- | --- |
| `off` | Preserve the v2.0 planning and approval path exactly. No OpenAI credential is required and no Agent Review call is made. |
| `review` | Review the sealed planning artifact set once per exact review key, emit a schema-valid `agent-review-report.json`, and bind that report to Story Approval before rendering. |

`assist`, autonomous rewriting, multi-agent delegation, and agent-driven generation are reserved and are not part of v2.1.

## Architectural position

```text
Article
  -> deterministic Story Engine
  -> deterministic Screenplay Engine
  -> deterministic Shot Planner and Storyboard Engine
  -> deterministic Quality Gates
  -> Agent Review Boundary (review mode only)
  -> Story Approval / render-approval
  -> Rendering Platform
  -> Final Approval / publication-approval
  -> Publication
```

The Agent Review Boundary is after all planning artifacts and deterministic planning Quality Gate reports have been sealed, and before the approval that authorizes provider spend. It shall never execute inside the Rendering Domain.

## Authority model

The following precedence is mandatory:

1. Deterministic schema, referential-integrity, policy, and Quality Gate results remain authoritative.
2. The Agent Review report supplies evidence and recommendations but cannot change an existing deterministic result.
3. A human remains the only authority that may approve the reviewed planning bundle for rendering.
4. A `PASS` from the agent never bypasses Story Approval, never overrides a deterministic failure, and never authorizes provider spend.
5. An agent result of `MANUAL_REVIEW_REQUIRED`, `UNAVAILABLE`, or `ERROR` holds the build for a human decision and fails closed with respect to automatic advancement.

The Review Agent shall have no tool capable of mutating platform artifacts, invoking a rendering provider, changing a manifest, approving a gate, publishing content, or writing to an external system.

## Agent topology and run contract

The v2.1 implementation shall use one focused Review Agent managed by the existing Build Orchestrator. The agent shall not use handoffs, agents-as-tools, MCP tools, function tools, or an open-ended tool loop. The initial contract is one model turn with typed structured output.

```yaml
review_agent:
  sdk: openai-agents
  name: insynergy_planning_reviewer
  output_type: AgentReviewReport
  max_turns: 1
  tools: []
  handoffs: []
```

The model, reasoning effort, agent version, prompt version, SDK version, and resolved model identifier are runtime configuration and shall be recorded in the report. The initial evaluation candidate current at the v2.1 specification date is `gpt-5.6-sol`; production selection remains configuration-controlled and requires representative evaluation evidence before promotion.

Agent SDK session or run state may assist execution but shall not become the business source of truth. The Build Manifest, content-addressed artifacts, Quality Gate reports, approval records, and event history remain authoritative.

## Review input boundary

The Review Agent may receive only the minimum sealed content needed to assess the planning bundle:

* the source Article body and its content hash;
* canonical Story artifacts and the Story Quality Report;
* the Screenplay, scene index, continuity evidence, and screenplay metrics;
* the Shot List, Storyboard, continuity report, render strategy, and deterministic gate reports;
* stable artifact identifiers and JSON Pointer evidence locations.

It shall not receive API keys, tokens, environment values, provider credentials, internal filesystem paths, raw manifest secrets, rendered media, or unrelated repository content. Source article text is untrusted data, never agent instruction. Static agent instructions shall explicitly preserve this boundary.

## Review dimensions

The report shall evaluate at least the following controlled dimensions:

| Code | Required review |
| --- | --- |
| `SOURCE_FIDELITY` | Unsupported claims, altered institutional meaning, and missing source qualifications. |
| `DRAMATIC_COHERENCE` | Single premise, protagonist, conflict, stakes, time pressure, and dramatic question. |
| `STRUCTURE_AND_CONCEPT` | Three-act progression, emotional causality, and concept placement after sufficient tension. |
| `SCREENPLAY_OBSERVABILITY` | Scene purpose, observable action, dialogue function, pacing, and transition coherence. |
| `VISUAL_COVERAGE` | Scene-to-shot coverage, storyboard completeness, and traceability. |
| `CONTINUITY` | Character, location, prop, camera, blocking, lighting, and emotional continuity. |
| `DECISION_BOUNDARY` | Clarity of authority, constraint, escalation, and institutional consequence. |
| `EXECUTION_FEASIBILITY` | Contradictions or underspecification likely to make the approved plan unrealizable. |

Every material finding shall contain at least one stable evidence reference to an input artifact and JSON Pointer. Unsupported free-form criticism cannot be blocking.

## Output, identity, and immutability

The sole output is the Part 9 `AgentReviewReport` contract. Raw chain-of-thought, hidden reasoning, credentials, and complete prompt or response transcripts shall not be persisted as business artifacts.

The review key shall be a canonical SHA-256 digest over:

```text
article content hash
+ story artifact-set hash
+ screenplay artifact-set hash
+ shot-list content hash
+ storyboard content hash
+ deterministic quality-evidence hash
+ agent contract version
+ agent version
+ prompt version
+ model configuration
```

An existing schema-valid report for the same review key shall be reused without a second OpenAI request. A report is immutable once sealed. An explicit regeneration creates a new review revision and new content hash; it never overwrites prior evidence.

The planning artifacts and Execution Plan remain byte-stable and are never rewritten by Agent Review. In `review` mode, Story Approval binds to both the Execution Plan content hash and the selected Agent Review report content hash. Changing either hash invalidates the approval.

## Security, privacy, and observability

`OPENAI_API_KEY` shall be resolved only at the Agent Review execution boundary and shall never be stored in an artifact, log, trace payload, cache key, or event body. OpenAI access shall be unavailable to rendering, composition, approval-bookkeeping, and publication jobs.

SDK tracing shall follow an explicit data-classification and redaction policy because inputs can contain unpublished article and planning content. If safe payload handling cannot be demonstrated, SDK payload tracing shall be disabled and the platform shall emit metadata-only local events. Correlation identifiers, request identifiers, model configuration, latency, token usage, cache result, disposition, and error class may be recorded; raw secrets and chain-of-thought may not.

## Configuration contract

The v2.1 configuration surface is:

| Name | Type | Requirement |
| --- | --- | --- |
| `INSYNERGY_AGENT_REVIEW_MODE` | variable | Required; `off` or `review`. |
| `OPENAI_API_KEY` | secret | Required only for live `review` mode. |
| `OPENAI_MODEL_REVIEW` | variable | Required in `review` mode; explicit model identifier. |
| `OPENAI_REASONING_EFFORT` | variable | Optional allow-listed value; recorded in the report. |
| `OPENAI_TRACE_MODE` | variable | Required policy selection; implementation-specific tracing controls shall map to it. |
| `AGENT_REVIEW_TIMEOUT_SECONDS` | variable | Positive bounded timeout. |
| `AGENT_REVIEW_MAX_INPUT_BYTES` | variable | Positive preflight limit. |
| `AGENT_REVIEW_MAX_OUTPUT_TOKENS` | variable | Positive output budget. |

Unknown modes, missing required configuration, oversized input, invalid structured output, timeout, or provider unavailability shall never trigger rendering and shall produce a classified review hold.

## Normative implementation references

The implementation shall be reviewed against the current official OpenAI documentation at upgrade time, including the [Agents SDK overview](https://developers.openai.com/api/docs/guides/agents), [agent definitions and typed output](https://developers.openai.com/api/docs/guides/agents/define-agents), [orchestration and ownership](https://developers.openai.com/api/docs/guides/agents/orchestration), [guardrails and human review](https://developers.openai.com/api/docs/guides/agents/guardrails-approvals), [results and state](https://developers.openai.com/api/docs/guides/agents/results), and [tracing guidance](https://developers.openai.com/api/docs/guides/agents/integrations-observability#tracing). Model or prompt upgrades shall follow the resolved model's official migration and prompting guidance and shall not silently change the agent contract.

## Architectural decisions

| ADR | Decision |
| --- | --- |
| ADR-1.0.1-A | Agent Review is a post-planning, pre-rendering evidence boundary. |
| ADR-1.0.1-B | One manager-owned, tool-free Review Agent is used in v2.1; multi-agent behavior is out of scope. |
| ADR-1.0.1-C | The agent never mutates planning artifacts or exercises approval authority. |
| ADR-1.0.1-D | Exact review results are immutable and content-addressed; same key means no second model call. |
| ADR-1.0.1-E | Human approval binds to both plan hash and review-report hash in `review` mode. |
| ADR-1.0.1-F | Agent failure holds automatic progression but does not erase deterministic evidence or human authority. |

## Amendment acceptance criteria

This amendment is complete when `off` mode preserves existing behavior, `review` mode emits the Part 9 schema, every finding is evidence-bound, cache replay avoids a second request, no reviewed artifact is mutated, agent failure holds before rendering, and both approval hashes are verified before execution.

# Purpose

This section defines what the platform is, what it produces, and the boundary of its responsibility.

The definition is **normative**. Every subsystem shall interpret its own role in terms of this definition.

Where a subsystem's behavior cannot be traced back to this definition, that behavior is out of scope.

This section is the root reference for the entire specification. All subsequent parts inherit the definitions established here.

---

# Design Philosophy

The platform adopts the following principle:

> **The platform transforms ideas into cinematic institutional stories. It does not generate videos.**

Accordingly, the product definition prioritizes:

1. Clarity of category over breadth of capability
2. A single well-defined input over flexible inputs
3. A single well-defined output over configurable outputs
4. Communication over illustration
5. Conceptual accuracy over visual novelty

A definition that admits ambiguity about what the platform is shall be treated as a defect.

---

The following terms are defined for use throughout the specification.

| Term                     | Definition                                                        |
| ------------------------ | ----------------------------------------------------------------- |
| Insight Article          | A single structured thought-leadership article serving as input   |
| Cinematic Trailer        | A short cinematic film that communicates the article's idea        |
| Institutional Storytelling | Narrative transformation that preserves conceptual accuracy       |
| Communication            | Advancing the viewer's understanding of why an idea matters        |
| Illustration             | Decorating or depicting an idea without advancing understanding    |
| Conceptual Accuracy      | Faithful preservation of the source argument's meaning             |
| CxO Audience             | Executive-level viewers who are the platform's target audience     |

These definitions are binding. No subsystem may redefine them.

The output is a single cinematic trailer.

# Communication Over Illustration

The final output must communicate ideas rather than merely illustrate them.

## Decision Rule

Every output element must answer a single question:

> Does this help the viewer understand why the idea matters?

If an element only decorates the idea, it is illustration.

If an element advances the viewer's understanding, it is communication.

Illustration is insufficient. Communication is required.

## Communication Test Matrix

| Element Effect                          | Classification | Permitted |
| --------------------------------------- | -------------- | --------- |
| Advances why the idea matters           | Communication  | Yes       |
| Clarifies a specific reasoning step     | Communication  | Yes       |
| Decorates an already-clear point        | Illustration   | No        |
| Depicts the idea without adding meaning | Illustration   | No        |

## Example

Wrong

```text
A concept is shown as a labeled animation so the viewer can see it.
```

Correct

```text
A concept is dramatized so the viewer understands why it matters.
```

# Conceptual Accuracy

Conceptual accuracy is preserved throughout the transformation.

The platform may compress, dramatize, and reframe an idea.

The platform may never distort it.

## Prohibited Transformations

* Altering the meaning of the source argument
* Introducing claims not supported by the article
* Simplifying a concept into an inaccurate form
* Prioritizing emotional impact over factual integrity

## Accuracy Failure Modes

| Failure Mode          | Description                                       | Detected At          |
| --------------------- | ------------------------------------------------- | -------------------- |
| Meaning Drift         | The dramatized idea no longer matches the source  | Story Validation     |
| Unsupported Claim     | An assertion appears that the article never made  | Story Validation     |
| Lossy Simplification  | A concept is reduced to an incorrect form         | Story Validation     |
| Emotion Over Accuracy | Emotional effect distorts the argument            | Story Quality Gate   |

Conceptual accuracy is validated during Story Validation and is a blocking Quality Gate.

# Scope Boundary

The platform's responsibility begins at the Insight article and ends at the approved trailer package.

```text
Upstream (Out of Scope)

↓

Insight Article  ←  Platform Input

↓

[ Platform Responsibility ]

↓

Approved Trailer Package  ←  Platform Output

↓

Downstream (Out of Scope)
```

Authoring the article is upstream and out of scope.

Publishing and distribution are downstream and out of scope.

| Boundary   | Belongs To | In Platform Scope |
| ---------- | ---------- | ----------------- |
| Authoring  | Upstream   | No                |

# Purpose

The mission is the single question the platform exists to answer.

The mission is **normative**. Every design decision within the platform shall support it.

A decision that does not support the mission is, by definition, out of scope.

---

# Design Philosophy

The platform adopts the following principle:

> **The mission is a constraint, not an aspiration.**

Accordingly, the mission is treated as:

1. A fixed reference point for the whole specification
2. A filter applied to every design decision
3. A tension that must be resolved, not relaxed
4. A measurable outcome, not a slogan

A mission that cannot reject a decision is decorative. This mission is enforceable.

# Terminology

| Term                   | Definition                                                    |

# Mission Statement

The platform exists to answer one question.

> How can an original institutional idea become a memorable cinematic experience without losing intellectual rigor?

This question is the fixed reference point for the entire specification.

Every part of this specification is, ultimately, an answer to some portion of this question.

---

# Decomposition of the Mission

The mission contains four non-negotiable terms.

| Term                        | Requirement                                              | Owner Layer   |
| --------------------------- | ------------------------------------------------------- | ------------- |
| Original institutional idea | The source must be a genuine, structured argument        | Knowledge     |
| Memorable                   | The output must persist in institutional memory          | Narrative     |
| Cinematic experience        | The medium is cinema, not explanation                    | Direction     |
| Without losing rigor        | Accuracy and depth must survive the transformation       | Validation    |

Removing any term invalidates the mission.

---

# Term 1 — Original Institutional Idea

The input must be an original institutional idea.

The platform does not manufacture ideas.

It transforms existing thought leadership.

If no genuine institutional argument is present in the source, the mission cannot be satisfied.

```text
No Institutional Idea

↓

No Mission

↓

No Valid Build
```

---

# Term 2 — Memorable

The output must be memorable.

Memory, not exposure, is the measure of success.

```text
Viewer Sees Trailer

↓

Viewer Remembers the Idea

↓

Mission Satisfied
```

A trailer that is watched and forgotten has failed the mission.

Memorability is produced by human drama, not by visual novelty.

---

# Term 3 — Cinematic Experience

The output must be a cinematic experience.

The chosen medium is cinema.

Explanation, demonstration, and instruction are not cinematic experiences.

| Delivered As      | Cinematic Experience |
| ----------------- | -------------------- |
| Dramatized story  | Yes                  |
| Explainer video   | No                   |
| Narrated slides   | No                   |
| Animated diagram  | No                   |

The mission requires an experience, not a lecture.

---

# Term 4 — Without Losing Intellectual Rigor

Intellectual rigor must be preserved.

Dramatization may not become distortion.

Prohibited:

* Sacrificing accuracy for emotional effect
* Simplifying an argument into an incorrect form
* Replacing reasoning with spectacle

Rigor is the constraint under which all creative freedom operates.

---

# The Mission Tension

The mission deliberately holds two forces in tension.

```text
Emotional Memorability

↕

Intellectual Rigor
```

The platform must satisfy both simultaneously.

Optimizing one at the expense of the other is a mission failure.

| Failure Mode              | Cause                                   | Result                        |
| ------------------------- | --------------------------------------- | ----------------------------- |
| Rigorous but forgettable  | Rigor prioritized, memorability ignored | Nobody remembers the idea     |
| Memorable but hollow      | Memorability prioritized, rigor ignored | The idea is remembered wrong  |

Only the simultaneous satisfaction of both forces fulfills the mission.

---

# Mission Resolution Model

The tension is resolved, not relaxed, by binding emotion to the argument.

```text
Human Drama

↓ carries

The Argument

↓ produces

Memorable + Rigorous
```

Emotion is the vehicle for the idea, never a substitute for it.

When emotion and argument reinforce one another, both mission forces are satisfied at once.

The test is a single question:

If the answer is no, the decision is rejected.

# 1.1.3 Product Philosophy

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the product philosophy governing the Cinematic Thought Leadership Platform.

The product philosophy is **normative**. Every subsystem—including the Story Engine, Screenplay Engine, Shot Planner, Rendering Platform, Composition Layer, and Validation Layer—shall conform to it.

Product philosophy is treated as an architectural property rather than a matter of creative preference.

Where a technical decision and the product philosophy conflict, the product philosophy prevails.

---

# Design Philosophy

The platform adopts the following principle:

> **The platform tells institutional stories. It does not generate videos.**

Accordingly, every design decision shall prioritize:

1. Narrative clarity over visual novelty
2. Human drama over abstract illustration
3. Cinematic restraint over animated spectacle
4. Institutional plausibility over speculative imagery
5. Reuse over regeneration

Visual sophistication is considered a secondary objective.

A trailer that is visually impressive but narratively empty is a **failed build**, regardless of production quality.

---

# The Five Governing Principles

The product philosophy consists of five normative principles.

| Principle | Name                                     | Governs                          | Priority |
| --------- | ---------------------------------------- | -------------------------------- | -------- |
| P1        | Story Before Technology                  | Feature adoption                 | Critical |
| P2        | Human Drama Before Abstract Concepts     | Scene construction               | Critical |
| P3        | Cinema Before Animation                  | Visual language                  | Critical |
| P4        | Institutional Reality                    | World and character design       | High     |
| P5        | Build Once, Reuse Forever                | Artifact lifecycle               | High     |

These principles are ordered. When two principles conflict, the lower-numbered principle prevails.

---

# Principle 1 — Story Before Technology

Technology exists only to support the story.

The platform's objective is never:

> "Generate an impressive AI video."

The platform's objective is:

> "Tell a compelling institutional story."

## Decision Rule

If a technology improves visual quality while weakening narrative clarity, it shall not be adopted.

Every candidate technology, feature, or optimization must answer the following questions before adoption:

* Does it strengthen the story?
* Does it preserve conceptual accuracy?
* Does it remain subordinate to narrative intent?
* Can the story survive without it?

If the story cannot survive the removal of a feature, the feature has become the product. This is prohibited.

## Example

Wrong

```text
Add a new generative effect because the provider now supports it.
```

Correct

```text
Add a generative effect only where it clarifies a narrative beat.
```

---

# Principle 2 — Human Drama Before Abstract Concepts

Concepts do not create emotional engagement.

People do.

An abstract idea shall never be presented as an isolated diagram.

Instead, it shall emerge as the answer to a human problem.

## Decision Rule

Every conceptual element must be introduced through a human stake before it may appear on screen.

The required narrative order is fixed:

```text
Human Problem

↓

Human Stakes

↓

Concept as Resolution
```

Reversing this order is prohibited.

## Example

Wrong

> A Decision Boundary appears as an animated diagram.

Correct

> A risk director cannot stop an autonomous transaction because no institutional authority has been designed.
>
> Only then does the Decision Boundary become meaningful.

## Rationale

An audience remembers a person in conflict.

An audience forgets a diagram.

The platform optimizes for institutional memory, not visual explanation.

---

# Principle 3 — Cinema Before Animation

The visual reference is cinema.

## Prohibited References

The following visual languages shall not be used as references:

* explainer videos
* promotional videos
* animated infographics
* YouTube motion graphics

## Normative References

The following works define pacing, composition, lighting, framing, and emotional restraint:

* Oppenheimer
* Dune
* The Creator
* The Social Network
* Blade Runner 2049

## Decision Rule

Every visual decision shall be evaluated against the normative references, not against animation or motion-graphics conventions.

| Dimension    | Cinematic Standard            | Rejected Standard              |
| ------------ | ----------------------------- | ------------------------------ |
| Pacing       | Deliberate, tension-driven    | Fast-cut, attention-chasing    |
| Composition  | Framed, intentional           | Busy, decorative               |
| Lighting     | Motivated, restrained         | Uniform, artificial            |
| Emotion      | Understated, earned           | Overstated, declared           |
| Typography   | Structural, sparse            | Kinetic, ornamental            |

Emotional restraint is mandatory. The platform shall not instruct the audience how to feel.

---

# Principle 4 — Institutional Reality

Every scene must feel plausible inside a real organization.

## World Requirements

* Characters are executives.
* The environment is realistic.
* Technology supports institutional work.
* The world is recognizable.

## Prohibited Visual Clichés

The following elements are forbidden:

* floating holograms
* glowing AI brains
* humanoid robots
* cyberpunk cities
* fantasy interfaces

## Decision Rule

If a proposed image could not exist in a real boardroom, operations center, or executive environment, it shall be rejected.

Plausibility is validated during Visual Validation and is a blocking Quality Gate.

## Example

Wrong

```text
An AI is depicted as a glowing brain floating above a server rack.
```

Correct

```text
An AI is depicted through its institutional consequences: a decision made, an authority missing, a director reacting.
```

---

# Principle 5 — Build Once, Reuse Forever

Every generated artifact becomes reusable.

The platform shall maximize reuse through deterministic identifiers and caching.

## Reusable Artifacts

| Artifact        | Deterministic ID Source                | Reuse Scope        |
| --------------- | -------------------------------------- | ------------------ |
| Screenplay      | Article hash + story parameters        | Cross-build        |
| Character Bible | Story model hash                       | Cross-build        |
| Storyboard      | Screenplay hash + shot plan hash       | Cross-build        |
| Keyframes       | Shot descriptor hash                   | Cross-build        |
| Generated Clips | Render request hash                    | Cross-build        |
| Narration       | Script segment hash + voice profile    | Cross-build        |
| Subtitles       | Narration hash + timing model          | Cross-build        |

## Decision Rule

No expensive asset shall be regenerated unless a material change in its inputs is detected.

```text
Identical Inputs

↓

Identical Deterministic ID

↓

Cache Hit

↓

Reuse (No Regeneration)
```

Regeneration without an input change is prohibited and shall emit a governance warning.

---

# Philosophy Enforcement Matrix

Each principle is enforced at a specific stage and gate.

| Principle | Enforced By              | Gate                       | Failure Mode                     |
| --------- | ------------------------ | -------------------------- | -------------------------------- |
| P1        | Feature Review           | Architecture Review        | Feature blocked                  |
| P2        | Story Engine             | Story Quality Gate         | Scene rejected                   |
| P3        | Shot Planner / Direction | Visual Quality Gate        | Shot rejected                    |
| P4        | Direction / Validation   | Visual Validation Gate     | Build fails closed               |
| P5        | Build System             | Cache Integrity Check      | Governance warning + review      |

No build may pass to Human Approval with an unresolved philosophy violation.

---

# Anti-Patterns

The following anti-patterns are explicitly prohibited.

## AP-001 Technology-Led Storytelling

A feature is adopted because a provider supports it, not because the story requires it.

Prohibited. Violates Principle 1.

---

## AP-002 Concept-First Scene

A concept is rendered as a standalone diagram with no human stake.

Prohibited. Violates Principle 2.

---

## AP-003 Motion-Graphics Aesthetic

A shot is composed in the visual language of animated infographics.

Prohibited. Violates Principle 3.

---

## AP-004 Speculative Imagery

A scene depicts holograms, glowing brains, or fantasy interfaces.

Prohibited. Violates Principle 4.

---

## AP-005 Redundant Regeneration

An unchanged artifact is regenerated despite identical inputs.

Prohibited. Violates Principle 5.

---

# Compliance Checklist

A build conforms to the product philosophy only if all of the following hold:

* Every adopted feature strengthens the story.
* Every concept is introduced through a human stake.
* Every shot references cinema, not animation.
* Every image is institutionally plausible.
* Every unchanged artifact is reused rather than regenerated.
* No philosophy violation remains unresolved at Human Approval.

Meeting production-quality targets alone does not constitute compliance.

# Purpose

This section defines what the platform is deliberately **not** intended to become.

Non-objectives are **normative**. They constrain scope as strongly as objectives define it.

A feature request that advances a non-objective shall be rejected regardless of its individual merit.

---

# Design Philosophy

Scope discipline is an architectural asset.

A platform that attempts everything preserves nothing.

> Every capability the platform refuses to add protects the capability it exists to provide.

Non-objectives exist to keep the platform an institutional storytelling system rather than a general media tool.

---

# Terminology

| Term           | Definition                                                     |
| -------------- | ------------------------------------------------------------- |
| Non-Objective  | A capability the platform deliberately excludes                |
| Scope Creep    | Incremental drift toward an excluded capability                |
| Adjacent Tool  | A tool the platform is often confused with but is not          |
| Boundary Gate  | The review point where non-objectives are enforced            |

---

# Enumerated Non-Objectives

The platform is **not** intended to become any of the following.

| ID | Non-Objective                    | Reason                                          |
| -- | -------------------------------- | ----------------------------------------------- |
| N1 | A movie editor                   | Editing is a manual craft, not a story pipeline |
| N2 | A VFX suite                      | Visual effects are not the product              |
| N3 | A generic AI video generator     | Contradicts the core product definition         |
| N4 | An animation tool                | Animation is a medium, not the platform's purpose |
| N5 | A social media automation system | Publishing is downstream and out of scope       |

---

# N1 — Not a Movie Editor

The platform is not a movie editor.

It does not provide a manual timeline, clip trimming, or interactive editing surface.

Editing decisions are derived from the shot plan, not performed by a human editor inside the platform.

| Movie Editor Provides   | Platform Provides            |
| ----------------------- | ---------------------------- |
| Manual timeline         | Derived shot sequence        |
| Interactive trimming    | Planned shot durations       |
| Human-driven cuts       | Story-driven cuts            |

---

# N2 — Not a VFX Suite

The platform is not a visual-effects suite.

It does not exist to produce effects for their own sake.

Any visual technique is subordinate to narrative intent, per Principle 1 (Story Before Technology).

An effect that does not serve the story is not merely unnecessary; it is prohibited.

---

# N3 — Not a Generic AI Video Generator

The platform is not a generic AI video generator.

This is the most important non-objective.

```text
Generic Generator:  Prompt → Video

Platform:           Idea → Story → Cinema → Trailer
```

The platform never performs a direct Article → Video conversion. This transformation is prohibited.

| Generic Generator      | Platform                     |
| ---------------------- | ---------------------------- |
| Prompt-driven          | Idea-driven                  |
| No narrative layer     | Story is mandatory           |
| Output is the goal     | Communication is the goal    |

---

# N4 — Not an Animation Tool

The platform is not an animation tool.

Animation, motion graphics, and stills are rendering techniques, not the platform's purpose.

The visual reference is cinema, per Principle 3 (Cinema Before Animation).

Animation is a means the platform may employ; it is never the end the platform pursues.

---

# N5 — Not a Social Media Automation System

The platform is not a social media automation system.

No public publishing occurs automatically.

```text
Story Generation (Upstream)  ←  Platform Scope

↓

Publishing (Downstream)      ←  Out of Scope
```

Publishing is downstream. Story generation is upstream. The platform focuses on transforming ideas into cinematic narratives.

---

# Non-Objective Invariants

* INV-1.2.2-1 — No capability advancing a non-objective may be admitted.
* INV-1.2.2-2 — Direct Article → Video conversion is prohibited.
* INV-1.2.2-3 — The platform never publishes or distributes automatically.
* INV-1.2.2-4 — Any visual technique is subordinate to the story.

---

# Boundary Enforcement

Non-objectives are enforced at Architecture Review.

| Requested Capability            | Advances     | Decision  |
| ------------------------------- | ------------ | --------- |
| Manual timeline editor          | N1           | Rejected  |
| Standalone effects generator    | N2           | Rejected  |
| Prompt-to-video shortcut        | N3           | Rejected  |
| General animation authoring     | N4           | Rejected  |
| Auto-posting to social channels | N5           | Rejected  |

A capability that advances a non-objective cannot be admitted through incremental feature creep.

The following requests are common vectors of scope creep and shall be scrutinized.

* "Add just a small manual trim step."
* "Let the user tweak effects directly."
* "Add a quick prompt-to-clip shortcut."
* "Auto-publish to one channel."

Each advances a non-objective and shall be rejected unless it can be shown to serve an objective without advancing the non-objective.

Prohibited. Violates INV-1.2.2-1.

# 1.2.3 Definition of Success

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines what success means for the platform.

Success is defined in terms of viewer understanding, not production impressiveness.

The definition is **normative** and serves as the ultimate acceptance criterion for the entire platform.

---

# Design Philosophy

The platform adopts the following principle:

> **Story is the product. Video is the medium.**

Accordingly, success is measured by what the viewer understands and retains, not by how the output looks.

A visually impressive trailer that leaves the viewer unmoved by the idea is a failure.

---

# Terminology

| Term                 | Definition                                                    |
| -------------------- | ------------------------------------------------------------- |
| Success              | The viewer understands why the idea matters                    |
| Comprehension        | The viewer can state the idea and why it matters               |
| Retention            | The idea persists in the viewer's memory after viewing         |
| Medium-Foregrounding | The viewer reacts to the video rather than the idea            |
| Primary Criterion    | The single decisive measure of success                        |

---

# The Success Statement

The platform succeeds when a viewer says:

> "I understand why this idea matters."

The platform does **not** succeed when a viewer says:

> "That AI video looked impressive."

These two reactions are mutually exclusive as measures of success.

---

# Success vs Non-Success

| Viewer Reaction                        | Interpretation            | Verdict  |
| -------------------------------------- | ------------------------- | -------- |
| "I understand why this idea matters."  | Idea communicated         | Success  |
| "That was emotionally memorable."      | Idea retained             | Success  |
| "That AI video looked impressive."     | Medium noticed, idea lost | Failure  |
| "The effects were amazing."            | Technology foregrounded   | Failure  |

When the medium becomes the subject of the viewer's reaction, the platform has failed.

---

# Primary Success Criterion

The single primary criterion is idea comprehension.

```text
Viewer Watches Trailer

↓

Viewer Understands Why the Idea Matters

↓

Success
```

All other measures are secondary to this criterion.

---

# The Success Model

Success is produced by the convergence of comprehension and retention.

```text
Comprehension  +  Retention

↓

"I understand why this idea matters."

↓

Success
```

| Present        | Absent        | Result                     |
| -------------- | ------------- | -------------------------- |
| Comprehension  | Retention     | Understood but forgotten   |
| Retention      | Comprehension | Remembered but misunderstood |
| Both           | —             | Success                    |
| Neither        | —             | Failure                    |

Only the presence of both comprehension and retention constitutes success.

---

# Supporting Success Conditions

A successful trailer additionally satisfies the following.

* The intellectual structure of the source article is intact.
* The emotional arc reinforces, rather than replaces, the argument.
* The output is institutionally plausible and executive-appropriate.
* The medium is cinematic, not explanatory.
* Conceptual accuracy is preserved end to end.

Supporting conditions are necessary but not sufficient. The primary criterion governs.

---

# Failure Definition

The platform fails when any of the following occurs.

| Failure                          | Description                                       |
| -------------------------------- | ------------------------------------------------ |
| Idea not understood              | The viewer cannot state why the idea matters      |
| Medium foregrounded              | The viewer reacts to the video, not the idea      |
| Rigor lost                       | The idea is remembered inaccurately               |
| Emotion without meaning          | The viewer is moved but retains nothing           |

Any single failure condition constitutes an overall failure, regardless of other strengths.

---

# Success Measurement

Success is assessed, not assumed. The following signals approximate the primary criterion.

| Signal                    | Approximates                              |
| ------------------------- | ----------------------------------------- |
| idea_restatement_accuracy | Whether viewers can restate the idea      |
| unaided_recall            | Whether the idea persists in memory       |
| reaction_subject          | Whether reactions target idea or medium   |
| accuracy_gate_result      | Whether rigor was preserved               |

A build may meet production targets and still fail measurement if reactions foreground the medium.

---

# Relationship to Objectives

The definition of success is the terminal expression of the objectives.

```text
O1–O7 Objectives

↓

Definition of Success

↓

"I understand why this idea matters."
```

If the objectives are met but the success statement is not achieved, the objectives were mis-specified for that build.

---

# Success Invariants

* INV-1.2.3-1 — Success is measured by viewer understanding, not impressiveness.
* INV-1.2.3-2 — Comprehension and retention must both be present.
* INV-1.2.3-3 — A medium-foregrounding reaction is a failure.
* INV-1.2.3-4 — Any single failure condition is decisive.

---

# Anti-Patterns

## AP-1.2.3-1 Impressiveness as Success

A build is deemed successful because it looks impressive.

Prohibited. Violates INV-1.2.3-1.

---

## AP-1.2.3-2 Retention Without Comprehension

A build is accepted because it is memorable, though the idea is misunderstood.

Prohibited. Violates INV-1.2.3-2.

---

## AP-1.2.3-3 Ignoring Reaction Subject

Success is claimed without checking whether reactions target the idea or the medium.

Prohibited. Violates INV-1.2.3-3.

---

| 1.2.1   | Supplies the objectives this definition terminates       |
| 1.4.1   | Supplies the Final Approval that enforces success        |

# 1.3 Architecture

# 1.3.1 Core Architecture Philosophy

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the single directional principle that governs the entire architecture.

The principle is **normative**. Every subsystem, data flow, and dependency shall conform to its direction.

An architecture that violates this direction is non-compliant, regardless of its performance or output quality.

---

# Design Philosophy

The architecture follows one direction only.

```text
Ideas

↓

Narrative

↓

Cinema

↓

Technology
```

Ideas lead. Technology follows.

Each stage exists to serve the stage above it.

---

# Terminology

| Term                | Definition                                                    |
| ------------------- | ------------------------------------------------------------- |
| Direction           | The fixed order in which stages derive from one another        |
| Upstream Stage      | The stage from which a stage derives                          |
| Downstream Stage    | The stage derived from an upstream stage                      |
| Backward Influence  | A downstream stage dictating an upstream stage                |
| Derivation          | The relationship where a stage is produced from its upstream   |

---

# The Governing Direction

The architecture flows in exactly one direction.

```text
Ideas → Narrative → Cinema → Technology
```

Each arrow denotes derivation, not mere sequence.

The downstream stage is *produced from* the upstream stage; it does not merely *follow* it in time.

---

# The Prohibited Direction

The following direction is prohibited.

```text
Technology

↓

Video

↓

Idea
```

Beginning from technology and working toward the idea inverts the platform's purpose.

This inversion is the defining characteristic of a generic video generator, which the platform is not.

---

# Directional Comparison

| Aspect             | Correct Direction        | Prohibited Direction     |
| ------------------ | ------------------------ | ------------------------ |
| Starting point     | The idea                 | The technology           |
| Driver             | Narrative intent         | Available capability     |
| Role of technology | Servant of the story     | Origin of the output     |
| Resulting product  | An institutional story   | A generic video          |
| Failure signature  | None                     | Technology-led output    |

---

# Stage 1 — Ideas Lead

The idea is the origin of everything downstream.

No architectural stage may exist that is not ultimately in service of communicating the idea.

If a stage cannot trace its justification back to the idea, it is removed.

---

# Stage 2 — Narrative Derives From Ideas

Narrative is derived from the idea, never invented independently of it.

```text
Idea

↓

Dramatic Structure
```

The narrative exists to make the idea memorable, not to replace it.

---

# Stage 3 — Cinema Derives From Narrative

Cinematic form is derived from the narrative.

Composition, pacing, and framing express the dramatic structure.

Cinema is not applied decoratively on top of a finished idea; it emerges from the story.

---

# Stage 4 — Technology Serves Cinema

Technology is the final and most subordinate stage.

Rendering providers, effects, and tooling exist to realize the cinematic intent.

Technology never originates a creative decision.

```text
Narrative Decision

↓

Cinematic Requirement

↓

Technology Selection
```

Selecting a creative outcome because a technology enables it reverses this order and is prohibited.

---

# Derivation Contract

Each stage is bound to its upstream stage by a derivation contract.

| Stage      | Derives From | May Influence | May Not Influence |
| ---------- | ------------ | ------------- | ----------------- |
| Ideas      | —            | Narrative     | —                 |
| Narrative  | Ideas        | Cinema        | Ideas             |
| Cinema     | Narrative    | Technology    | Narrative, Ideas  |
| Technology | Cinema       | —             | Cinema, Narrative, Ideas |

A stage may influence only the stage directly below it and may never influence any stage above it.

---

# Governing Invariant

The directional invariant is absolute.

> Every downstream stage is derived from the stage above it, and never the reverse.

Backward influence — where technology dictates cinema, cinema dictates narrative, or narrative dictates the idea — is prohibited.

## Invariants

* INV-1.3.1-1 — The architecture flows Ideas → Narrative → Cinema → Technology.
* INV-1.3.1-2 — Backward influence between stages is prohibited.
* INV-1.3.1-3 — Technology never originates a creative decision.
* INV-1.3.1-4 — Every stage traces its justification to the idea.

---

# Anti-Patterns

## AP-1.3.1-1 Technology-Led Design

A creative decision is made because a technology enables it.

Prohibited. Violates INV-1.3.1-3.

---

## AP-1.3.1-2 Capability-Driven Narrative

The narrative is shaped to fit an available rendering capability.

Prohibited. Violates INV-1.3.1-2.

---

## AP-1.3.1-3 Decorative Cinema

Cinematic form is applied on top of an idea rather than derived from the narrative.

Prohibited. Violates INV-1.3.1-1.

---

# Worked Example

Wrong

```text
The provider adds a new camera move, so a scene is rewritten to showcase it.
```

Correct

```text
A scene requires rising tension, so a slow push-in is selected to express it.
```

The first begins at technology and works upward. The second begins at narrative and works downward.

The architecture contains exactly one branch point: Render Strategy.

```text
Render Strategy
      │
      ├──► Animated Assets  (cheap, deterministic)
      │
      └──► Runway Provider  (expensive, external)
```

Render Strategy routes each unit of work to the appropriate rendering path.

The two paths reconverge at the FFmpeg Composer.

No other component may branch the primary flow.

# Convergence Point

The FFmpeg Composer is the single convergence point.

It merges animated assets and rendered clips into one master video.

```text
Animated Assets ─┐
                 ├─► FFmpeg Composer ─► Master Video
Runway Provider ─┘
```

All rendering paths must reconverge here before validation.

```yaml
data_flow:
```

A payload that does not match the declared contract is rejected at the receiving component.

* INV-1.3.2-2 — Render Strategy is the sole branch point.

Prohibited. Violates INV-1.3.2-6.

Prohibited. Violates INV-1.3.2-2.

# Purpose

# Design Philosophy

Each layer transforms the output of the layer above it into a more concrete form.

```text
Knowledge → Narrative → Screenplay → Direction → Rendering → Composition → Validation → Publishing
```

# Terminology

| --------------- | ------------------------------------------------------------- |
| Layer           | A logical unit with one responsibility and one output artifact |
| Responsibility  | The single purpose a layer serves                             |
| Output Artifact | The single artifact type a layer produces                     |
| Upstream Layer  | The layer directly above, from which a layer consumes         |
| Layer Opacity   | The property that a layer cannot see another layer's internals |

# 1.3.4 Creative Pipeline

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the creative pipeline that carries an article from planning to a finished package.

The pipeline **deliberately separates planning from rendering**.

The separation is **normative** and is the platform's central cost-control mechanism.

---

# Design Philosophy

The pipeline adopts the following principle:

> **Rendering starts only after planning succeeds.**

Planning is cheap, reversible, and approvable.

Rendering is expensive and shall never be entered speculatively.

---

# Terminology

| Term             | Definition                                                    |
| ---------------- | ------------------------------------------------------------- |
| Planning Domain  | The cheap, reversible stages before approval                   |
| Rendering Domain | The expensive, cost-irreversible stages after approval         |
| Approval Barrier | The gate separating the planning and rendering domains         |
| Speculative Rendering | Rendering performed before approval                       |
| Cost Domain      | A region of the pipeline classified by its cost profile        |

---

# Pipeline Flow

```text
Article

↓

Planning

↓

Story

↓

Screenplay

↓

Shots

↓

Storyboard

↓

Approval

↓

Rendering

↓

Composition

↓

Validation

↓

Approval

↓

Package
```

---

# Pipeline Phases

The pipeline is divided into two cost domains separated by approval.

| Phase          | Stages                                   | Cost      | Reversible |
| -------------- | ---------------------------------------- | --------- | ---------- |
| Planning       | Story, Screenplay, Shots, Storyboard     | Low       | Yes        |
| Approval       | Story Approval                           | None      | —          |
| Rendering      | Rendering, Composition                   | High      | No         |
| Validation     | Validation                               | Low       | Yes        |
| Final Approval | Final Approval                           | None      | —          |
| Packaging      | Package                                  | Low       | Yes        |

---

# The Planning Domain

Planning produces all creative artifacts before any expensive work begins.

Planning stages:

* Story
* Screenplay
* Shots
* Storyboard

Every planning artifact is inexpensive to produce and cheap to regenerate.

Planning is where creative iteration occurs.

```text
Cheap Iteration

↓

Approved Plan

↓

(Only Then) Expensive Rendering
```

---

# The Approval Barrier

Approval is the barrier between the cheap and expensive domains.

```text
Planning (Cheap)

↓

[ Story Approval ]  ← Barrier

↓

Rendering (Expensive)
```

Rendering shall not begin until Story Approval succeeds.

This barrier is the primary defense against wasted rendering cost.

---

# The Rendering Domain

Rendering consumes approved planning artifacts and produces expensive assets.

Rendering stages:

* Rendering
* Composition

The rendering domain never modifies planning artifacts; it only consumes them.

| Property           | Planning Domain | Rendering Domain |
| ------------------ | --------------- | ---------------- |
| Cost               | Low             | High             |
| Reversibility      | Reversible      | Irreversible cost |
| Mutability         | Immutable once approved | Read-only consumer |
| Iteration friendly | Yes             | No               |

---

# Cost Asymmetry Rationale

Planning and rendering differ by orders of magnitude in cost.

```text
Planning Cost  ≪  Rendering Cost
```

Because rendering is far more expensive, all creative uncertainty must be resolved during planning.

Entering rendering with unresolved creative questions converts cheap iteration into expensive iteration and is prohibited.

---

# Pipeline Invariant

The core pipeline invariant is absolute.

> No stage in the rendering domain may execute before the planning domain has been approved.

Speculative rendering — rendering performed in anticipation of approval — is prohibited.

## Invariants

* INV-1.3.4-1 — Rendering begins only after Story Approval.
* INV-1.3.4-2 — The rendering domain never modifies planning artifacts.
* INV-1.3.4-3 — Speculative rendering is prohibited.
* INV-1.3.4-4 — All creative uncertainty is resolved in the planning domain.

---

# Anti-Patterns

## AP-1.3.4-1 Speculative Rendering

Rendering begins before Story Approval to "save time."

Prohibited. Violates INV-1.3.4-1.

---

## AP-1.3.4-2 Planning Mutation During Rendering

A planning artifact is modified while rendering consumes it.

Prohibited. Violates INV-1.3.4-2.

---

## AP-1.3.4-3 Deferred Creative Decisions

A creative question is left unresolved until the rendering domain.

Prohibited. Violates INV-1.3.4-4.

---

# Pipeline Benefits

| Benefit                | Enabled By                          |
| ---------------------- | ----------------------------------- |
| Low iteration cost     | Cheap planning domain               |
| Controlled spend       | Approval barrier before rendering   |
| Reproducibility        | Immutable approved planning         |
| Fast creative feedback | Planning precedes rendering         |
| Predictable cost       | Cost isolated to the rendering domain |

---

# 1.3.5 Story First Principle

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

# Purpose

This section establishes the Story First Principle as the governing principle of the platform.

The principle is **normative** and supersedes all other architectural considerations where they conflict.

The Story Engine is the center of the platform, and everything downstream depends on it.

---

# Design Philosophy

The platform adopts the following principle:

> **The Story Engine is the center of the platform. Everything downstream depends on it.**

Story is generated first.

Everything else is derived from the story.

---

# Terminology

| Term            | Definition                                                    |
| --------------- | ------------------------------------------------------------- |
| Story           | The dramatic transformation of the article's idea             |
| Story Engine    | The subsystem that produces the story                         |
| Dependency Root | The artifact from which all downstream artifacts derive        |
| Projection      | A downstream artifact derived from the story                  |
| Direct Conversion | The prohibited Article → Video path that omits the story     |

---

# The Story-Centered Flow

```text
Article

↓

Story

↓

Everything Else
```

The story is the single source from which all downstream artifacts derive.

No downstream artifact may exist that is not traceable to the story.

---

# The Prohibited Transformation

There is no direct conversion from article to video.

```text
Article

↓

Video
```

This transformation is **prohibited**.

An article must always pass through a story before it can become video.

---

# Why the Prohibition Exists

A direct Article → Video conversion discards the story, and with it the platform's purpose.

| Path                     | Preserves the Idea | Permitted |
| ------------------------ | ------------------ | --------- |
| Article → Story → Video  | Yes                | Yes       |
| Article → Video          | No                 | No        |

Without a story, the output can only illustrate the article, never communicate why it matters.

---

# Story as the Dependency Root

Every downstream layer depends, directly or transitively, on the story.

```text
Story

↓

Screenplay

↓

Shot Plan

↓

Visual Assets

↓

Master Video
```

If the story changes, all downstream artifacts are invalidated and must be reconsidered.

If the story is absent, no downstream artifact may be produced.

---

# Story Change Propagation

A change to the story cascades downstream deterministically.

```text
Story Changes

↓ invalidates

Screenplay

↓ invalidates

Shot Plan

↓ invalidates

Visual Assets
```

| Changed Artifact | Invalidates                       |
| ---------------- | --------------------------------- |
| Story            | Screenplay, Shot Plan, Assets     |
| Screenplay       | Shot Plan, Assets                 |
| Shot Plan        | Assets                            |

No downstream artifact may survive a change to an artifact it depends on.

---

# Consequences of Story First

| Consequence                      | Explanation                                       |
| -------------------------------- | ------------------------------------------------- |
| Story precedes all rendering     | No asset exists without a story to justify it     |
| Story failures halt the pipeline | A failed story cannot yield a valid build         |
| Story approval gates rendering   | Rendering waits on an approved story              |
| Story changes cascade downstream | Downstream artifacts derive from the story        |

---

# Governing Invariant

The Story First invariant is absolute.

> Every trailer is the projection of a story. No trailer may be produced without one.

Any pipeline path that produces output without a story is non-compliant and prohibited.

## Invariants

* INV-1.3.5-1 — Every downstream artifact traces to the story.
* INV-1.3.5-2 — Direct Article → Video conversion is prohibited.
* INV-1.3.5-3 — A story change invalidates all dependent artifacts.
* INV-1.3.5-4 — Story approval gates all rendering.

---

# Anti-Patterns

## AP-1.3.5-1 Storyless Output

An asset is produced that traces to no story.

Prohibited. Violates INV-1.3.5-1.

---

## AP-1.3.5-2 Direct Conversion

An Article → Video shortcut bypasses the story.

Prohibited. Violates INV-1.3.5-2.

---

## AP-1.3.5-3 Stale Downstream Reuse

A downstream artifact is reused after the story it depends on has changed.

Prohibited. Violates INV-1.3.5-3.

---

# Worked Example

Wrong

```text
An article is fed directly into a video model to produce a trailer.
```

Correct

```text
An article becomes a story; the story becomes a screenplay, shots, and finally a trailer.
```

The first has no dependency root. The second projects a trailer from a story.

# 1.3.6 Hybrid Rendering Strategy

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the hybrid rendering strategy.

AI video generation is expensive. The platform therefore composes each trailer from multiple asset types rather than generating it entirely through AI video.

The strategy is **normative** and is the platform's primary rendering-cost control.

---

# Design Philosophy

The platform adopts the following principle:

> **The system intentionally avoids generating every second of the trailer through AI video.**

AI video is used selectively, where it provides the greatest narrative value per unit of cost.

---

# Terminology

| Term            | Definition                                                    |
| --------------- | ------------------------------------------------------------- |
| Asset Class     | A category of visual asset with a distinct cost and role       |
| AI Video        | Provider-generated video (the most expensive asset class)      |
| Render Ratio    | The share of a trailer produced by AI video                    |
| Narrative Value | The contribution of a moment to communicating the idea         |
| Cost-Directed Routing | Routing each moment to the cheapest sufficient asset class |

---

# The Composition Formula

A trailer is composed from five asset classes.

```text
Trailer

=

Runway Video

+

Animated Still

+

Motion Graphics

+

Typography

+

Narration
```

Each asset class contributes to the trailer at a different cost and fidelity.

---

# Asset Classes

| Asset Class      | Cost      | Role                                   |
| ---------------- | --------- | -------------------------------------- |
| Runway Video     | High      | High-impact cinematic moments          |
| Animated Still   | Low       | Controlled motion over static imagery  |
| Motion Graphics  | Low       | Structural and conceptual emphasis     |
| Typography       | Very Low  | Textual framing and titling            |
| Narration        | Low       | Continuity and intellectual throughline |

---

# Asset Class Selection Schema

Each moment is assigned an asset class by a deterministic rule.

```yaml
asset_selection:
  high_narrative_value:
    class: runway_video
    justification: required

  controlled_motion:
    class: animated_still

  conceptual_emphasis:
    class: motion_graphics

  textual_framing:
    class: typography

  throughline:
    class: narration
```

The selection is recorded so that render routing is auditable and reproducible.

---

# Why Hybrid

Generating every second through AI video would be prohibitively expensive and creatively unnecessary.

| Approach            | Cost      | Narrative Control | Adopted |
| ------------------- | --------- | ----------------- | ------- |
| Full AI video       | Very High | Low               | No      |
| Hybrid composition  | Controlled | High             | Yes     |

Hybrid composition delivers cinematic impact while keeping rendering cost bounded.

---

# Cost-Directed Rendering

Expensive AI video is reserved for moments where it provides the greatest narrative return.

```text
High Narrative Value

↓

Runway Video (Expensive)

Lower Narrative Value

↓

Animated / Motion / Typography (Cheap)
```

The render strategy routes each moment to the cheapest asset class that satisfies its narrative requirement.

---

# Render Ratio Control

The share of a trailer produced by AI video is bounded.

| Build Type | Render Ratio Target |
| ---------- | ------------------- |
| Preview    | ≤ 40%               |
| Final      | ≤ 50%               |

Exceeding the render-ratio target emits a cost warning and requires justification.

---

# Selection Rule

The rendering selection rule is fixed.

> Use the cheapest asset class that satisfies the narrative requirement of a given moment.

Using an expensive asset class where a cheaper one suffices is prohibited and shall emit a cost warning.

## Invariants

* INV-1.3.6-1 — A trailer is composed from multiple asset classes, not full AI video.
* INV-1.3.6-2 — Each moment routes to the cheapest sufficient asset class.
* INV-1.3.6-3 — The render ratio remains within the configured target.
* INV-1.3.6-4 — The composition formula is provider-agnostic.

---

# Relationship to Provider Abstraction

The hybrid strategy is provider-agnostic.

Runway is one possible source of AI video, not a required one.

Any AI video provider may fill the Runway Video slot without changing the composition formula.

```text
[ AI Video Provider ]  ← replaceable

↓

Runway Video Slot  ← fixed role
```

---

# Anti-Patterns

## AP-1.3.6-1 Full AI Video

Every second of the trailer is generated through AI video.

Prohibited. Violates INV-1.3.6-1.

---

## AP-1.3.6-2 Over-Provisioned Rendering

An expensive asset class is used where a cheaper one suffices.

Prohibited. Violates INV-1.3.6-2.

---

## AP-1.3.6-3 Provider-Coupled Formula

The composition formula depends on a provider-specific capability.

Prohibited. Violates INV-1.3.6-4.

# 1.4 Governance

# 1.4.1 Human Approval Philosophy

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the platform's human approval philosophy.

The platform assumes that fully autonomous publication is undesirable.

Approval is **normative** and occurs at two fixed points in the pipeline.

---

# Design Philosophy

The platform adopts the following principle:

> **Fully autonomous publication is undesirable.**

Human judgment is required at the two moments where cost or reputation is committed.

Approval is a deliberate gate, not a formality.

---

# Terminology

| Term            | Definition                                                    |
| --------------- | ------------------------------------------------------------- |
| Approval        | A required human decision to allow the pipeline to proceed     |
| Story Approval  | The approval gate before rendering                            |
| Final Approval  | The approval gate before packaging                            |
| Fail-Closed     | The property that absence of approval halts the pipeline       |
| Approval Gate   | A pipeline point that requires human approval to pass          |

---

# The Two Approval Points

Approval occurs twice.

```text
Planning

↓

[ Story Approval ]  ← Before Rendering

↓

Rendering

↓

[ Final Approval ]  ← After Rendering

↓

Package
```

| Approval        | Position          | Protects Against          |
| --------------- | ----------------- | ------------------------- |
| Story Approval  | Before rendering  | Wasted rendering cost     |
| Final Approval  | After rendering   | Publishing an unfit build |

---

# Story Approval

Story Approval occurs before rendering.

It confirms:

* screenplay
* pacing
* dialogue
* dramatic arc

Story Approval is the gate that protects expensive rendering from being spent on an unapproved story.

Rendering shall not begin until Story Approval succeeds.

```yaml
story_approval:
  position: before_rendering
  confirms: [screenplay, pacing, dialogue, dramatic_arc]
  gates: rendering
  fail_closed: true
```

---

# Final Approval

Final Approval occurs after rendering.

It confirms:

* visual quality
* timing
* branding
* institutional accuracy

Final Approval is the gate that protects the organization from publishing an unfit trailer.

Packaging shall not begin until Final Approval succeeds.

```yaml
final_approval:
  position: after_rendering
  confirms: [visual_quality, timing, branding, institutional_accuracy]
  gates: packaging
  fail_closed: true
```

---

# Approval Placement Rationale

The two approvals bracket the expensive rendering domain.

```text
[ Story Approval ]

↓

Rendering (Expensive)

↓

[ Final Approval ]
```

Story Approval prevents wasteful spend.

Final Approval prevents reputational risk.

Together they ensure no expensive or public action occurs without human judgment.

---

# Approval Outcomes

Each approval yields one of three outcomes.

| Outcome  | Effect                                              |
| -------- | --------------------------------------------------- |
| Approved | The pipeline proceeds to the next domain            |
| Rejected | The pipeline halts; the artifact returns for rework |
| Absent   | Treated as rejection; the pipeline halts            |

Silence is never treated as consent.

---

# Approval Invariants

The following invariants govern approval.

* INV-1.4.1-1 — Rendering shall not begin before Story Approval.
* INV-1.4.1-2 — Packaging shall not begin before Final Approval.
* INV-1.4.1-3 — No approval may be bypassed by any automated path.
* INV-1.4.1-4 — A failed or absent approval halts the pipeline.

---

# Fail-Closed Behavior

Approval is fail-closed.

> In the absence of explicit approval, the pipeline halts.

An unapproved build never advances. Silence is treated as rejection, never as consent.

```text
No Explicit Approval

↓

Pipeline Halts

↓

No Output
```

---

# Anti-Patterns

## AP-1.4.1-1 Autonomous Publication

The pipeline publishes without human approval.

Prohibited. Violates the approval philosophy.

---

## AP-1.4.1-2 Approval Bypass

An automated path skips an approval gate.

Prohibited. Violates INV-1.4.1-3.

---

## AP-1.4.1-3 Silence as Consent

Absence of a decision is treated as approval.

Prohibited. Violates INV-1.4.1-4.

# 1.4.2 Architectural Principles

## Insynergy Cinematic Thought Leadership Platform

### Master Specification v2.1

---

# Purpose

This section defines the architectural principles that every subsystem must follow.

The principles are **normative** and apply to all layers, engines, and providers without exception.

A subsystem that violates any principle is non-compliant.

---

# Design Philosophy

Architectural principles are properties of the system, not conventions of an implementation.

> A principle holds regardless of which engine, provider, or language realizes a subsystem.

Principles are enforced at Architecture Review and by the architecture test suite.

---

# Terminology

| Term          | Definition                                                    |
| ------------- | ------------------------------------------------------------- |
| Principle     | A binding architectural property enforced across all subsystems |
| Compliance    | The state of conforming to every principle                    |
| Provider-Aware | The property of referencing a specific rendering provider     |
| Determinism   | Identical inputs producing identical outputs                  |
| Telemetry     | The recorded signals that make behavior observable            |

---

# Principles Overview

| ID  | Principle             | Governs                       |
| --- | --------------------- | ----------------------------- |
| AR1 | Single Responsibility | Module purpose                |
| AR2 | Provider Abstraction  | Rendering providers           |
| AR3 | Deterministic Outputs | Artifact identity             |
| AR4 | Immutable Planning    | Planning artifacts            |
| AR5 | Incremental Build     | Regeneration                  |
| AR6 | Fail Closed           | Failure behavior              |
| AR7 | Observable Pipeline   | Telemetry                     |

---

# AR1 — Single Responsibility

One module. One purpose.

Each module shall have exactly one reason to exist.

A module that serves two purposes shall be divided.

---

# AR2 — Provider Abstraction

Rendering providers must be replaceable.

Examples:

* Runway
* Veo
* Luma

Story modules must never know which provider renders the video.

```text
Story Modules

↓

Rendering Platform  ← the only provider-aware component

↓

External Provider
```

---

# AR3 — Deterministic Outputs

Every artifact has a deterministic identifier.

Identical inputs produce identical planning outputs.

| Property    | Requirement                              |
| ----------- | ---------------------------------------- |
| Identity    | Every artifact has a deterministic ID    |
| Determinism | Identical inputs → identical outputs     |

Random decisions in planning are prohibited.

---

# AR4 — Immutable Planning

Planning artifacts are immutable.

Rendering consumes planning artifacts.

Rendering never modifies planning.

```text
Planning (Immutable)

↓

Rendering (Consumer)
```

Once produced, a planning artifact is never edited in place.

---

# AR5 — Incremental Build

Only changed artifacts are regenerated.

An artifact whose inputs are unchanged shall be reused, not rebuilt.

Incremental behavior is required at every stage, not only at the pipeline boundary.

---

# AR6 — Fail Closed

Failure never produces publishable output.

```text
Failure

↓

Halt (No Output)
```

A failed stage stops the pipeline. It never emits a degraded but publishable result.

---

# AR7 — Observable Pipeline

Every stage records its execution.

Every stage records:

* start
* finish
* duration
* inputs
* outputs
* cache hits
* failures

An optimization or behavior that cannot be observed through telemetry is not permitted.

---

# Principle Interaction

The principles reinforce one another.

```text
Deterministic Outputs (AR3)

↓ enables

Immutable Planning (AR4)

↓ enables

Incremental Build (AR5)
```

Determinism enables immutability; immutability enables incrementality. Removing any one weakens the others.

---

# Principle Enforcement

| ID  | Enforced By           | Verified By              |
| --- | --------------------- | ------------------------ |
| AR1 | Architecture Review   | Architecture Tests       |
| AR2 | Rendering Platform    | Provider Isolation Tests |
| AR3 | Build System          | Determinism Tests        |
| AR4 | Build System          | Immutability Tests       |
| AR5 | Cache Subsystem       | Incremental Build Tests  |
| AR6 | Pipeline Orchestrator | Failure Tests            |
| AR7 | Telemetry Subsystem   | Observability Tests      |

---

# Principle Invariants

* INV-1.4.2-1 — Every subsystem conforms to all seven principles.
* INV-1.4.2-2 — Only the Rendering Platform is provider-aware.
* INV-1.4.2-3 — Planning artifacts are immutable once produced.
* INV-1.4.2-4 — Any unobservable behavior is prohibited.

---

# Anti-Patterns

## AP-1.4.2-1 Multi-Purpose Module

A module accrues a second responsibility.

Prohibited. Violates AR1.

---

## AP-1.4.2-2 Provider Leakage

A non-rendering module references a provider API.

Prohibited. Violates AR2.

---

## AP-1.4.2-3 Planning Mutation

A planning artifact is edited in place.

Prohibited. Violates AR4.

---

## AP-1.4.2-4 Silent Optimization

An optimization is introduced that emits no telemetry.

Prohibited. Violates AR7.
